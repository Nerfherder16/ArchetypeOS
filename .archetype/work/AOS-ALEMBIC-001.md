## Design

- **Scaffold** `apps/api/alembic/` with `env.py` that: imports `Base` and `app.models` (so all 20 tables register on `Base.metadata`), reads the URL from `get_settings().database_url` (NOT a hardcoded ini URL), sets `target_metadata = Base.metadata`, and runs online migrations with `compare_type=True`.
- **Baseline migration** `0001_baseline.py`: autogenerate against a FRESH empty DB → `op.create_table(...)` for all 20 tables (DDL rendered from model metadata, so portable sqlite↔postgres); `downgrade()` drops them. `down_revision = None`.
- **Startup wiring**: `docker-entrypoint.sh` = `set -e; alembic upgrade head; exec uvicorn app.main:app --host 0.0.0.0 --port 8000`. A failed migration exits non-zero → container never starts uvicorn → `/health` never responds → compose-smoke catches it (migration failures are surfaced, never masked). Dockerfile copies `alembic.ini` + `alembic/` and sets the entrypoint. `init_db()` create_all stays: on fresh postgres alembic runs first (entrypoint, before uvicorn), so create_all is a no-op; it only builds schema for sqlite dev where no entrypoint runs.
- **Discipline doc**: `docs/DATABASE_MIGRATIONS.md` — how to autogenerate a migration, the review rule (inspect autogenerated DDL before commit), and the one-time `alembic stamp head` for pre-existing databases (teevee-1's populated DB must be stamped, not upgraded, so the baseline isn't re-applied).

## Acceptance Criteria

- Alembic scaffolded and wired to the models — evidence: `alembic/env.py` targets `Base.metadata` with all 20 tables; `alembic.ini` present.
- Baseline creates the full current schema — evidence: on a fresh sqlite DB, `alembic upgrade head` creates all 20 tables + `alembic_version` (Orchestrator asserts table count); `alembic current` shows the baseline revision.
- Round-trip clean — evidence: `alembic downgrade base` drops all baseline tables without error.
- Container runs migrations before serving — evidence: `docker-entrypoint.sh` runs `alembic upgrade head` then `exec uvicorn`; Dockerfile copies alembic files + uses the entrypoint; the migration hard-fails the container on error.
- No schema drift vs models — evidence: after `alembic upgrade head`, `alembic revision --autogenerate` produces an EMPTY migration (no pending changes) — proving the baseline matches the models exactly.
- Nothing else broken — evidence: full pytest (67 API + 1 worker) green unchanged; ruff/compileall exit 0.
- Postgres path proven — evidence: CI compose-smoke green on the PR (fresh postgres, entrypoint runs the baseline).
- Discipline documented — evidence: `docs/DATABASE_MIGRATIONS.md` incl. the `stamp head` runbook step.
