# LES-029 — Reusing one pinned branch across multiple PRs (restarted from `origin/main` after each merge) guarantees local/remote divergence and forces a force-push every cycle — which auto-mode blocks

## Aliases

- pinned branch reuse forces a force-push
- restart-from-main after merge diverges the branch
- auto-mode blocks force-push on a reused branch
- one fresh branch per PR off latest main
- GitHub auto-delete head branches on merge

## Status

open

## Owner

Chief Architect / Orchestrator

## Evidence

- This remote session ran AOS-EMBED-001/002, AOS-UI-001, and the reconciliation passes all off a single env-pinned branch (`claude/aos-runtime-002-scanner-1egyjw`), restarting it from `origin/main` after each merge. Because the remote head still carries the previous PR's commits while the local branch is reset to the new `main` tip, the two histories diverge every cycle — so the next push is a non-fast-forward that only a force-push resolves, and auto-mode (the non-interactive push path) refuses a force-push. Each cycle then needs a manual force-push or an out-of-band reset.
- Root cause: a branch is a moving pointer, not a workspace. Reusing the same branch name for sequential PRs and hard-resetting it to `main` after each merge overwrites history the remote already has; the remote can't fast-forward to an older-rooted tip. The env-pinned branch-name constraint (Decision 2a) is what makes the remote session reuse the name rather than cut a new one.
- Compounding: with the branch reused, GitHub never deletes the merged head (there is nothing to delete — the name persists), so stale merged commits linger on the reused head and widen the divergence the next reset must overwrite.

## Content

- Event: the same env-pinned branch was reused for sequential PRs and restarted from `origin/main` after every merge, so local and remote diverged each cycle and the next push required a force-push that auto-mode blocks.
- Source: this remote session's multi-PR run on `claude/aos-runtime-002-scanner-1egyjw`.
- Category: process.
- Lesson: **one fresh branch per PR, cut off the latest `main`** — never reuse a branch across PRs and never hard-reset a branch that the remote already has commits on. Keep post-merge reconciliation **inside the feature PR** (so a separate reconciliation push onto the same reused branch is not needed), and enable GitHub **"auto-delete head branches on merge"** so a merged head cannot linger and re-diverge. Where a session is env-pinned to a fixed branch name (Decision 2a), that pin is in direct tension with this practice — either lift the pin so each PR can cut a new branch, or accept a scripted force-with-lease reset as the explicit, reviewed exception (auto-mode will still block it, so it must be a manual step).
- Loop feed: **open** — the env-pinned branch-name constraint (Decision 2a) still forces reuse in the remote session, so the "fresh branch per PR" fix is not yet operationalized here; the actionable items are (1) resolve the env-pin vs fresh-branch tension and (2) turn on auto-delete head branches on merge.
- Status: open — cite this row when a push fails as a non-fast-forward on a reused branch, or when planning branch strategy for a multi-PR session.
