## Evidence

Verified on `main` implementing RFC-0013 Slices 2-4 (Slice 1 extraction already shipped):

- **The evidence existed but was inert.** Distillation's reasoned tier already NAMED the
  reusable capabilities a different project could borrow (`{name, description, provenance}`,
  grounded to cited files, capped) — but `distill_repository` only rendered them into the
  vault markdown and **persisted nothing**. The DB carried the one-sentence `DNA.purpose`
  and a `title + purpose` embedding; the named capabilities never reached a table, so no
  query could match them. Extraction without persistence is documentation, not a feature.
- **Reuse was failing on granularity, not on lexical-vs-semantic.** The 5-repo shakedown
  proved a reuse need matched against `DNA.purpose` is near-useless both lexically ("LLM
  provider abstraction" → no match) and semantically ("agent framework" → ~0.24 noise, the
  actual agent system not in the top 3). A product-level purpose ("an engineering
  intelligence platform") answers "what IS this repo?"; a reuse need asks "what CAPABILITY
  inside it can I borrow?" — a component-level question product-level evidence cannot
  answer at any similarity threshold. Adding a better scorer to the wrong-granularity
  corpus could not have fixed it.

## Content

- **Event:** capability evidence was extracted (Slice 1) but never persisted or matched;
  reuse recommendations stayed noise because the searchable corpus was at product
  granularity while the question is at component granularity.
- **Category:** self-found-defect (found while wiring the persistence + matching path).
- **Lesson (transferable):**
  1. **Persist extracted evidence at the granularity you will QUERY it.** A capability
     named in prose but not stored as a row is unqueryable. Give each reusable unit its own
     `RepositoryCapability` row (`repository_id`/`knowledge_page_id`/`name`/`description`/
     `provenance`/`embedding`) and embed `name + " " + description` per capability — a need
     then matches a *single* capability's vector (high cosine), not a whole-product blob
     (noise). The granularity of the stored unit IS the ceiling on match quality.
  2. **When retrieval returns noise, check the granularity of the corpus before the scorer.**
     Lexical→semantic is a *precision* lever; product→component is a *granularity* lever. The
     shakedown ruled out the former (both failed) and named the latter. Diagnose which axis is
     broken before spending effort on the other.
  3. **Replace-only-when-produced, so a cheap re-run never destroys rich data.** The capability
     set is a pure function of the current source, so a reasoned re-distill replaces the rows
     wholesale — but a *deterministic* re-distill (the hermetic CI provider, empty narrative)
     must leave prior reasoned rows untouched, exactly like the page embedding is only
     overwritten when a real embedder produced one. A no-op tier must never clobber.
  4. **Make the new path a strict superset.** Gate the capability path on "any capability
     exists" and fall back byte-for-byte to the repo-level floor when none were extracted (the
     deterministic tier) or the need matches none — so sqlite/CI behaviour is provably
     unchanged and there is no regression while the actionable path is added.
  5. **Cite the unit + its file, not just the repo.** A recommendation that names the specific
     capability and its provenance file ("reuse X's *LLM provider-routing abstraction* —
     `core/llm_router.py`") is actionable; a bare repo pointer is not. That is the payoff of
     storing provenance per capability.
- **Loop feed:** the `RepositoryCapability` table + per-capability embeddings are the substrate
  for a capability-cited Reuse UI surface and for capability-to-capability duplicate detection
  across the portfolio (follow-ups). The RFC-0013 §8 portfolio benchmark
  (`scripts/reality_test_capability_reuse.py`) is the standing regression gate for extraction +
  matching quality on real repos.
