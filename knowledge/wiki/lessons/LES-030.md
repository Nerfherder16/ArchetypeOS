# LES-030 — Section-scoping isn't enough when one section renders two lists of the same entity

## Content

- Event: `decisionRow` was hardened by LES-028 to scope to the "Decisions & Research" `section` (excluding the separate AOS-COUNCIL-002 dashboard). But that same section renders TWO lists that both echo the question: the Decision-Loop's own council-reviews list (`councilReviews.map`, each row shows `· {question}` + a "Draft decision" button) AND the decisions list (`decisions.map`). After drafting, the section-scoped `getByRole('listitem').filter({ hasText: question })` matched both → strict-mode violation.
- Source: PR #81 CI (self-found on the first CI run of an unrelated CI/docs PR).
- Category: ci-failure.
- Lesson: scoping a Playwright locator to the owning section (LES-028) is necessary but not sufficient — when a single section renders more than one list of the same entity, `listitem + dynamic-text` is still ambiguous. Prefer a **stable `data-testid` on the specific rows you target** (`data-testid="decision-row"`) over structural `getByRole('listitem')` filters; the testid survives co-located lists, re-orderings, and new sibling surfaces. This is the pattern LES-028 already noted the new COUNCIL-002 spec got right (`data-testid="council-review-row"`).
- Fix: added `data-testid="decision-row"` to the decision `<li>` in `main.tsx`; `decisionRow` is now `page.getByTestId('decision-row').filter({ hasText: question })`. Verified: `npm run build` clean, `decision-loop.spec.ts` green 3/3 consecutive local runs.
- Loop feed: fixed in the same change set as AOS-OPS-001 (the CI-green-signal PR that surfaced it).
- Status: closed.
