# LES-031 — CI Guardian reads the PR body from the event payload; edit the body BEFORE the triggering push

## Aliases

- missing-acceptance-evidence after a late body edit
- PR body edit does not re-trigger CI
- finalize PR body before pushing code paths

## Status

## Evidence

- PR #81 CI (run 28835761476, job 85519117957): PR Guardian `BLOCK: missing-acceptance-evidence` on head `a9d8c0c`, even though the local guardian PASSED on that exact head and the PR body on GitHub already had an `## Acceptance Evidence` section. The commit added `apps/web/src/main.tsx` (newly triggering the acceptance-evidence gate), and the body was updated *after* the push.

## Content

- Event: pushed a commit that first introduced an `apps/web/src/**` change (which makes the guardian require `## Acceptance Evidence`), then updated the PR body to add that section via the API. CI Guardian still BLOCKed.
- Source: PR #81 CI (self-found).
- Category: ci-failure.
- Lesson: the CI Guardian job reads the PR body from `github.event.pull_request.body` — the snapshot captured when the `pull_request` (synchronize/opened) event fired. **Editing the PR body afterward does NOT re-run the workflow**, and re-running the *same* run re-reads the stale event payload, so the fix is not "re-run" — it is a **new** push (synchronize) so a fresh payload carries the current body. Practical rule: when a commit adds a code path that newly trips a body-dependent gate (Acceptance Evidence, Verification Metadata), finalize the PR body **before** pushing that commit; if the body was edited after the push, re-trigger with a new commit (an empty commit works, but a meaningful one — e.g. recording this lesson — is better).
- Fix / loop feed: re-triggered #81 with the commit that adds this lesson, so the new synchronize event carries the already-updated body (with Acceptance Evidence) and the guardian re-reads it.
- Status: closed.
