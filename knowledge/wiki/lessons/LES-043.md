# LES-043 — PR Guardian's `missing-verification-metadata` needs six PLAIN `Field: value` lines; an `## Acceptance Evidence` section does not satisfy it, and a body edit alone does not re-run the check

## Aliases

- missing-verification-metadata BLOCK despite having Acceptance Evidence
- Verification Status / Verification Level / Verification Method / Evidence / Limitations / Required Next Verifier
- `- **Evidence:**` bold-bullet wrapper does not parse
- Guardian re-reads the PR body on push, not on body-edit

## Status

validated

## Owner

Chief Architect / Orchestrator (cloud session)

## Evidence

- PR #221 (AOS-COUNCIL-VALIDATION-MODELS-001) carried a full `## Acceptance Evidence` section
  (bullet lines each containing the literal `evidence:` token — satisfying the *separate*
  `missing-acceptance-evidence` check, LES-041) but PR Guardian still BLOCKed on
  `missing-verification-metadata`: "PR body is missing required verification metadata fields:
  Verification Status, Verification Level, Verification Method, Evidence, Limitations, Required
  Next Verifier. Fields must be plain `Field: value` lines at the start of a line; markdown
  bold/bullet wrappers (e.g. `- **Field:** value`) do not parse."
- `tools/pr_guardian.py`: `check_verification_metadata` requires all six
  `REQUIRED_VERIFICATION_FIELDS`; `field_value` matches `(?im)^\s*<Field>\s*:\s*(.+?)\s*$` — a
  line-start `Field: value` (leading whitespace ok), NOT a `-`/`*` bullet or `**bold**` wrapper.
  `Verification Status` must be one of `ALLOWED_VERIFICATION_STATUSES`; `Verification Level` one
  of `Level 0..5`; `Verification unavailable`/`Verification blocked` also block merge. The four
  free-text fields fire a *weak-verification-metadata* WARN (not BLOCK) if too thin.
- The `## Acceptance Evidence` check (`missing-acceptance-evidence`) and the verification-metadata
  check are DISTINCT gates with DIFFERENT formats: the former wants `-`/`*` bullets containing
  `evidence:`; the latter wants plain `Field: value` lines. A PR needs BOTH.
- Re-trigger: Guardian runs off the pushed body snapshot, so fixing the body then pushing a commit
  (here, this lesson file) re-runs it; a bare body edit does not (LES-041).

## Content

- Event: an implementation PR that already carried an `## Acceptance Evidence` section still
  BLOCKed on `missing-verification-metadata`, because that is a separate check demanding a
  six-field plain-text block, not the bullet Evidence list.
- Rules:
  1. **Every PR body needs the six plain `Field: value` verification lines** — `Verification
     Status:`, `Verification Level:`, `Verification Method:`, `Evidence:`, `Limitations:`,
     `Required Next Verifier:` — at line start, no `-`/`*`/`**bold**` wrappers (they don't parse).
     Copy the block from `docs/VERIFICATION_PROTOCOL.md` §"PR Metadata".
  2. **It is a DIFFERENT gate from `## Acceptance Evidence`.** Keep both: bullets-with-`evidence:`
     for acceptance (LES-041), plain `Field: value` for verification metadata. One does not
     satisfy the other.
  3. **Use an allowed, non-merge-blocking status.** `Verification unavailable`/`Verification
     blocked` block merge; when CI is still the real gate, use `Verified with warnings` (or
     `Verification pending`) at the local level reached (e.g. Level 2), name the honest limitation,
     and set `Required Next Verifier: GitHub CI`.
  4. **Fix the body, then PUSH** — Guardian reads the pushed snapshot, not a bare body edit
     (LES-041). Pre-stage this block (with LES-040's capability-map and LES-041's docs +
     Acceptance Evidence) for every queued PR so all three routine gates pass on first open.
