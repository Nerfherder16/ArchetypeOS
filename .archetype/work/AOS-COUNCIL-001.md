## Design

### Provider abstraction (`aos_core/llm/`)
- `ProviderResult` (dataclass): `text: str`, `provider: str`, `model: str | None`, `finish_reason: str`.
- `Provider` (typing.Protocol): `generate(self, *, system: str, prompt: str, max_tokens: int = 1024) -> ProviderResult`.
- `DeterministicProvider`: real, shippable, offline. `generate` returns a stable, structured **JSON** `text` derived only from `(system, prompt)` — no randomness, no seed, no clock. It is what CI runs. The seam is what matters — a real model drops in behind `Provider` unchanged.
- `ClaudeCodeProvider`: the operator's real backend. `generate` invokes the local Claude Code SDK / headless CLI (`claude -p <prompt> --output-format json`, prepending the system prompt) via `subprocess`/SDK, using the **subscription** auth already on the node — **no keys read from repo config, nothing committed**. Returns the model's text in `ProviderResult`. Selected only when `llm_provider="claude_code"` on an authed node; **never invoked in CI**. Tested with the subprocess/SDK boundary **mocked** (assert it builds the right argv/prompt and maps stdout→`ProviderResult`; a non-zero exit / missing binary → a clear error). Keep the invocation thin and defensive (timeout, decode, error surface).
- `get_provider(settings) -> Provider`: `deterministic` → `DeterministicProvider()`; `claude_code` → `ClaudeCodeProvider()`; unknown → a clear `ValueError`. Contract mirrors the Verification Provider Interface (declare → standardized result); Constitution Art. XI (Modular Intelligence).
- **Structured-output contract**: persona prompts request a JSON object (`summary`/`findings`/`evidence`/`concerns`/`confidence`/`status`); the council service parses it **tolerantly** — unparseable prose → `status="Needs Evidence"`, low confidence, raw text in `summary`. Both providers flow through the one parser.

### Council service (`aos_core/services/council.py`)
- `DEFAULT_AGENTS`: the four personas — `research_librarian`, `architecture_cartographer`, `technology_fitness_judge`, `security_agent` — each `{name, agent_type, system_prompt, evidence_selector}`.
- **Evidence substrate** (resolves RFC Open Question 2): each agent reads the project's latest scan / DNA / decisions already in the DB as its evidence; the deterministic provider transforms `(persona, evidence, question)` into a structured agent output. No external research step in the seed.
- `run_council(db, *, project_id, question, provider, agents=DEFAULT_AGENTS) -> CouncilReview`:
  1. 404-guard the project (raise if missing).
  2. For each agent: build the persona prompt over the evidence, call `provider.generate(...)`, parse into a `CouncilAgentOutput` — `status` (`Complete` / `Needs Evidence` / `Escalated` / `Rejected`), `summary`, `findings[]`, `evidence[]`, `concerns[]`, `confidence`.
  3. `synthesize_verdict(outputs)` (Final Judge): `agreements[]`, `disagreements[]` (explicitly surfaced — a finding two agents contradict), `unsupported_claims[]` (findings with empty evidence), aggregate `confidence`, `follow_up[]`, and a `verdict` from the documented set (`Accept` / `Accept with warnings` / `Reject` / `Defer` / `Research further` / `Simulate first` / `Escalate to human`). **Abstention rule**: if evidence/confidence below a documented floor (`ABSTAIN_CONFIDENCE = 0.35`, `MIN_EVIDENCE = 1`) → verdict `Insufficient evidence` with `follow_up` = what must be verified.
  4. Persist a `CouncilReview` + its `CouncilAgentOutput` rows; commit; return the review. Rule-based judge = deterministic + auditable (the Arbiter rules are code; the first-pass reasoning is the provider).

### Models + migration
- `CouncilReview(AuditMixin, Base)` table `council_reviews`: `project_id` (GUID FK, indexed), `question` (Text), `verdict` (String(64)), `confidence` (Float default 0.0), `agreements`/`disagreements`/`unsupported_claims`/`follow_up` (JSONField default list), `provider` (String(128)), `job_id` (GUID FK nullable).
- `CouncilAgentOutput(AuditMixin, Base)` table `council_agent_outputs`: `review_id` (GUID FK `council_reviews.id`, indexed), `agent_name` (String(128)), `agent_type` (String(128)), `status` (String(64)), `summary` (Text), `findings`/`evidence`/`concerns` (JSONField default list), `confidence` (Float default 0.0).
- Migration `0003_council` (`down_revision='0002'`): two `create_table`s; add `import aos_core.models` manually. Post-migration no-drift autogenerate probe MUST be 0 ops.

### Worker + API
- **Worker**: `elif job_type == "council_review":` → read `question`/`project_id` from the job (payload/columns), `provider = get_provider(settings)`, `review = run_council(db, project_id=…, question=…, provider=provider)`, set `review.job_id`, `mark_job(job_id, "completed", result={"review_id": review.id, "verdict": review.verdict})`.
- **API** (resolves RFC Open Question 1 — enqueue-primary, consistent with "control plane decides, nodes execute"):
  - `POST /projects/{project_id}/council-reviews` — body `{question}`; 404 if project missing; `enqueue_job(db, redis_client, job_type="council_review", project_id=…, payload={"question": …})`; returns the created `Job` (`JobRead`).
  - `GET /projects/{project_id}/council-reviews` — recent reviews (desc by created_at, cap 50).
  - `GET /council-reviews/{review_id}` — the review + nested agent outputs + verdict fields; 404 if missing.

## Acceptance Criteria

- **Four agents + Final Judge produce structured, persisted outputs** — evidence: `test_run_council` — `run_council` over a seeded project yields a `CouncilReview` with 4 `CouncilAgentOutput` rows, each with non-empty `summary`/`findings`/`status`/`confidence`, and a verdict from the documented set.
- **Disagreement is visible** — evidence: `test_council_surfaces_disagreement` — a constructed evidence set where agents diverge produces a non-empty `disagreements[]` on the review.
- **Judge abstains on insufficient evidence** — evidence: `test_council_abstains` — a project with no scan/DNA/decisions → verdict `Insufficient evidence` with a non-empty `follow_up[]`.
- **Provider is abstracted + deterministic** — evidence: `test_deterministic_provider_stable` — two `generate(...)` calls with equal inputs return equal `text`; `get_provider` returns a `DeterministicProvider` for `deterministic`, a `ClaudeCodeProvider` for `claude_code`, and raises on an unknown name.
- **Claude Code backend adapter is covered without a live call** — evidence: `test_claude_code_provider_mocked` — with the subprocess/SDK boundary mocked, `ClaudeCodeProvider.generate` builds the expected invocation (system+prompt) and maps stdout → `ProviderResult`; a non-zero exit / missing binary raises a clear error. No real `claude` call in CI.
- **Council runs as a job** — evidence: `apps/worker` `test_council_review_dispatch` — a `council_review` job runs `run_council` and `mark_job` completes with `{review_id, verdict}`.
- **API trigger + read** — evidence: `test_council_api` — `POST …/council-reviews` enqueues a job (job row + FakeRedis lpush; 404 on missing project); `GET …/council-reviews` and `GET /council-reviews/{id}` return the persisted review + agent outputs (seeded via `run_council`); 404 on missing review.
- **No schema drift** — evidence: `alembic upgrade head` creates `council_reviews` + `council_agent_outputs`; autogenerate no-drift probe → 0 ops after `0003`.
- **Advisory only** — evidence: no route sets `approved_by`; council output carries no execution side effect.
- **Nothing broken** — evidence: api suite (77 + new) green; worker suite (6 + 1) green; ruff/compile clean.
- **Postgres + migration** — evidence: CI compose-smoke green (migration `0003` applies on fresh Postgres via the `alembic upgrade head` entrypoint; all services boot).
