# LES-036 — Routing that isn't on the claim path is advisory; the claim's compare-and-swap is the only place assignment + eligibility are actually enforced

## Aliases

- advisory routing vs enforced routing
- global-queue bypass closed at the claim
- fail-open sensitivity (_rank → 0 = public)
- AOS-NODE-EXECUTION-001 closing lesson

## Status

closed

## Owner

Orchestrator (cloud session)

## Evidence

Verified on merged `main` implementing AOS-NODE-EXECUTION-001 (WP3; consumes [[LES-033]], builds on [[LES-034]] fencing):

- **Routing computed a decision nothing obeyed.** `route_job`/`node_eligibility` implemented the full eligibility predicate (capability ∈ caps ∧ sensitivity ≤ ceiling ∧ write ≤ policy ∧ fresh health) with a deterministic explanation — but its ONLY caller was the read-only `GET /nodes/route`. `enqueue_job` never routed, the `Job` row persisted no assignment, and the worker `brpop`'d one global queue and claimed by lease alone. The "capability-aware routing" capability existed as a pure function off the execution path — the exact [[LES-033]] pattern, one layer up from the fencing case.
- **Sensitivity failed OPEN.** `routing._rank` was `except ValueError: return 0` — an unknown/misspelled sensitivity collapsed to `public` (rank 0), so a typo'd `"secrett"` job passed every node's ceiling check. Sensitivity was an unvalidated free string with a ladder disjoint from the only `Sensitivity` enum in the tree.

## Content

- **Event:** capability-aware routing had a correct eligibility calculator and a nice explanation, but it gated nothing — a job ran on any worker that popped its id, and an unknown sensitivity was silently downgraded to public.
- **Category:** self-found-defect (found while wiring routing to execution, verified against source).
- **Lesson (transferable):**
  1. **A routing/eligibility decision is advisory until it is a predicate INSIDE the atomic claim.** Persist the decision (assigned node + requirements) at origination for audit, but enforce it where the row is actually taken: extend the claim's compare-and-swap WHERE with `assigned_node_id ∈ (NULL, this_node)` and re-validate eligibility against the *live* node at claim time (never trust the origination-time decision — health/capability drift). Then a mis-delivered id simply loses the CAS and is dropped; you do not need per-node queues to prevent a job running on the wrong node — the claim is the enforcement point. Delivery is an optimization; the claim is the boundary.
  2. **Derive a job's requirements server-side; never trust the client.** A `job_type → {capability, sensitivity, requires_write}` registry in the shared package (kept in agreement with the worker handlers by a test) is the authority. The client sends a job type, the server decides what running it requires.
  3. **Rank unknown enum values to FAIL CLOSED, and validate at the boundary.** `except ValueError: return 0` is fail-open — an unknown class becomes the *least* restrictive. Rank an unknown value ABOVE every real ceiling (so it is never eligible) and 422 it at the API. One shared, validated `Sensitivity` type, used by routing + origination + the API, kills the disjoint-ladder drift.
  4. **"No eligible node" is a visible waiting state, not a lost job.** Leave the job `queued`/`no_eligible_node` and let a reconciler sweep assign it when a node becomes eligible (and re-route jobs whose assigned node went stale); mid-execution node death is the WP1 lease reaper's job. Losing the job or silently routing it to an ineligible node are both wrong.
- **Loop feed:** the server-owned `job_requirements` registry is the seam AOS-AUTHORITY-HARDEN-001 (WP4) extends with `action_class` for server-owned action classification; the fail-closed `Sensitivity` type feeds WP4's repo-sensitivity-from-policy work.
- **Status:** closed — consumed by AOS-NODE-EXECUTION-001 (branch `claude/aos-node-execution-001`): Job routing fields + migration 0024, `claim_job_for_node` (assignment+eligibility CAS on the WP1 token), origination routing, `reroute_waiting_jobs`, shared fail-closed `Sensitivity`, multi-node PostgreSQL tests.
