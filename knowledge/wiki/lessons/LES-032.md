# LES-032 — An exception-swallowing instrumentation wrapper hides sink bugs; a ledger sink needs an end-to-end test, and factory defaults must merge with (never duplicate) forwarded kwargs

## Aliases

- silent instrumentation wrapper swallows TypeError
- ledger records zero events despite passing unit tests
- factory default kwarg collides with wrapper-forwarded kwarg
- best-effort sink hides its own bugs

## Status

validated

## Owner

Orchestrator (cloud session)

## Evidence

- AOS-USAGE-001 build (self-found): the `InstrumentedProvider` swallows any sink exception by design (a ledger failure must never break a real LLM call). The first `make_ledger_sink` passed `context=` as a factory-level default while the wrapper ALSO forwarded `context` in its per-call `fields` → `sink()` was called with `context` twice → `TypeError: got multiple values for keyword argument 'context'`. Because the wrapper swallows sink exceptions, this raised, was caught, and **recorded zero `UsageEvent`s** — while every unit test (which mocked the sink or asserted the wrapper returned the inner result) still passed. Caught only by an end-to-end sanity check (InstrumentedProvider → make_ledger_sink → real DB → summarize_usage returned 0 events).

## Content

- Event: an instrumentation wrapper that intentionally swallows sink exceptions masked a real defect (a duplicate-kwarg `TypeError`), so the ledger silently recorded nothing while the unit suite stayed green.
- Source: AOS-USAGE-001 (self-found during the build's end-to-end sanity check).
- Category: self-found-defect.
- Lesson:
  1. **Best-effort/exception-swallowing wrappers must be paired with an end-to-end test.** A wrapper that catches-and-continues (correct for resilience) also catches-and-hides its own wiring bugs. Unit tests that mock the sink or only assert transparency (inner result returned) cannot see a sink that always throws. The only test that catches it drives the *real* path: wrapper → real sink → real DB → read back the recorded row.
  2. **Factory-level defaults must MERGE with forwarded kwargs, never be passed alongside them.** When a closure both (a) captures default labels and (b) is invoked with a `**fields` dict that may carry those same keys, apply the default only when the forwarded value is absent/`None` (`fields.setdefault`-style), rather than passing the default positionally/explicitly on top of `**fields`. Passing both raises `TypeError: multiple values for keyword`.
- Fix / loop feed: `make_ledger_sink`'s inner `sink(**fields)` now fills `context`/`agent`/`session` from the factory defaults **only when `fields.get(...) is None`**, never double-passing; added regression test `test_make_ledger_sink_records_end_to_end` (drives wrapper → sink → sqlite → `summarize_usage`, asserts exactly one event with correct tier/cost/model). Verifier note: recorded per CLAUDE.md (a self-found defect requires a lesson in the same change set).
- Status: closed.
