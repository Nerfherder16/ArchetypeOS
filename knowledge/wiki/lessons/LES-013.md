# LES-013 — file-count language mix misreads a library's primary language on config/docs-heavy repos

## Aliases

- YAML-repo misclassification
- language_mix weighting gap
- primary-language false signal

## Status

closed

## Owner

Chief Architect / Orchestrator

## Evidence

- AOS-21 portfolio reality test (`docs/PORTFOLIO_PYDANTIC_AI.md`, `.archetype/portfolio/*/scan.json`): scanning `pydantic/pydantic-ai` — an unambiguous Python library — produced `language_mix` with YAML (1183 files) > Python (564), i.e. **Python only 28% by file count**. A second repo makes the gap precise: `anthropics/claude-agent-sdk-python` (a lean, source-heavy SDK) reads **correctly** at **77% Python** — so the misread is **repo-dependent**, biting on config/docs/CI-heavy real-world repos. `primary_language_hints` only partly compensates on the docs-heavy case. Any downstream read keyed on the top language (a naive Technology-Fitness / DNA summary) would misclassify the monorepo.

## Content

- Event: the repository scanner's `language_mix` counts files, so a config/docs/CI-heavy real-world repo ranks YAML/Markdown above the actual source language; the "primary" language signal is misleading on real code (surfaced the first time the system scanned a repo it did not write).
- Source: AOS-21 reality test, self-found on a real external repo.
- Category: self-found-defect
- Lesson: **raw file-count language mix is a misleading primary signal on real-world repos.** Rank by lines-of-code and/or classify source vs. config/docs/CI before deciding a repo's primary language. Until fixed, treat `language_mix`/`primary_language_hints` as file-distribution data, not a primary-language verdict — and do NOT let a Technology-Fitness / DNA summary key on the top entry alone.
- Loop feed: scoped follow-up package (source-classified language ranking) — shipped as `AOS-ARCH-SEMANTICS-001`. The scanner now classifies each language (`LANGUAGE_CLASS`: source/config/markup/data/docs), derives `primary_language` from the top **source** language (fallback: overall top), and ranks `primary_language_hints` source-first; `language_mix` raw counts are retained for backward compatibility. LoC-based weighting was rejected — the scanner does not read source bodies, and classification fixes the misread without new I/O.
- Status: closed — consumed by `AOS-ARCH-SEMANTICS-001` (source-classified language weighting + `primary_language`/`language_classes` in the scan summary, with tests on a docs/config-heavy fixture asserting `primary_language == "Python"`).
