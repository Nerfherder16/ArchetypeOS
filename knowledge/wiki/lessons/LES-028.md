## Content

- Event: AOS-COUNCIL-002 added a second Control Tower surface ("Agent Council") that renders the same council reviews the Decision Loop already lists. The existing `decision-loop.spec.ts` located its decision row with a page-global `getByRole('listitem').filter({ hasText: question })`, which had silently relied on the question text being unique to one listitem on the page. The new surface made it match two → strict-mode violation → the Web e2e job failed.
- Source: PR #74 CI (self-found on the first CI run).
- Category: ci-failure.
- Lesson: a page-global Playwright locator over dynamic text is fragile — the moment another surface renders the same entity it becomes ambiguous. Scope e2e locators to the section that owns the element (`page.locator('section').filter({ has: page.getByRole('heading', { name: '...' }) }).getByRole('listitem')...`), or key off a section-unique `data-testid`. Same meta-lesson as LES-025 (do not couple tests to global content), now for Playwright locators. The fix restores the exact pre-change behavior: the global locator matched exactly one element before (inside the Decisions section), so scoping to that section reproduces that match and excludes the new surface. Note the new AOS-COUNCIL-002 spec was itself unaffected because it keyed off a section-unique `data-testid="council-review-row"` — the right pattern for new surfaces.
- Secondary catch (process): this lesson was first authored as LES-027, but the remote session's AOS-EMBED-002 grabbed LES-027 for a `ci.yml` lesson in the same window (both cut from a main topping at LES-026). The `.gitattributes` union driver (LES-026) auto-merged the index but produced a duplicate-ID row and an add/add conflict on the file — union resolves additive line conflicts, not semantic ID collisions. Renumbered to LES-028 on merge; the remote's LES-027 stays canonical. Reinforces LES-008 (re-derive IDs from the live index at write time — and re-check after every merge).
- Loop feed: fixed in the same change set — `decisionRow` scoped to the "Decisions & Research" section.
- Status: closed.
