# LES-023 — A Jaccard over the candidate's whole vocabulary collapses confidence magnitudes; retrieval confidence should be *need coverage*

## Aliases

- transfer confidence too small to be meaningful
- Jaccard denominator dominated by the candidate's own tokens
- uncalibrated relevance score
- need-coverage confidence

## Status

closed

## Owner

Chief Architect / Orchestrator

## Evidence

- Full end-to-end reality test of the intelligence loop (Orchestrator, 2026-07-06), run over the 6-repo portfolio with the reasoned distillation tier (AOS-DISTILL-004): the Knowledge Transfer Engine returned the **correct repo #1 on every query with honest matched-term provenance**, but every confidence was tiny (0.01–0.13) — e.g. `kubernetes` #1 on "container orchestration" at `conf=0.0147`, `gin` #1 on "HTTP routing" at `conf=0.108`.
- Root cause (verified against source): `score_relevance` used `jaccard = |need ∩ cand| / |need ∪ cand|`. The union is dominated by the candidate's **own** vocabulary (a distilled `purpose` tokenizes to many terms), so a correct match still scores near zero. The magnitude was therefore an artifact of candidate verbosity, not of match quality — and misleading: a UI showing "confidence 0.09" for the #1 recommendation reads as a *bad* match, blocking the engine from being surfaced honestly in the product.

## Content

- Event: the shipped Transfer Engine MVP ranked correctly but reported uncalibrated, near-zero confidence, because the relevance score was a Jaccard over the candidate's full token set.
- Source: full end-to-end reality test on the real cloned portfolio, self-found.
- Category: design-insight.
- Lesson: **a retrieval confidence should measure how much of the *query* is satisfied, not token overlap against the candidate's whole vocabulary.** Coverage — `|(need ∩ cand) ∪ (need ∩ tech)| / |need|` — is bounded `0..1`, intuitive ("2 of 3 need terms → 0.667"), invariant to how verbose the candidate's distilled text is, and preserves (indeed sharpens) ranking. Jaccard penalises a rich, descriptive candidate exactly for being descriptive — the opposite of what a good distillation should earn.
- Loop feed: **consumed** — AOS-TRANSFER-002 (Package 3). `score_relevance` now returns need coverage; `recommend_reuse` breaks ties by technology-match count then name. On the real portfolio the confidences became meaningful (kubernetes "container orchestration" 0.333, gin "HTTP routing" 0.800) with rankings intact, and the "agent framework with tool calling" ranking *improved* — `pydantic-ai` (an actual agent framework, 2/4 covered) now correctly outranks the `claude-agent-sdk` wrapper (1/4). This also unblocks surfacing transfer in the product (calibrated confidence is a prerequisite for an honest UI).
- Status: closed — cite this row if a retrieval/relevance score is defined as overlap against a candidate's full vocabulary rather than as coverage of the query, or when adding the embeddings tier (keep the reported confidence coverage-like/calibrated, not a raw similarity).
