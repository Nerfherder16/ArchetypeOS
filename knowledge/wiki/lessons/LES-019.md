# LES-019 — A structural scan is the wrong evidence class for an adoption question; the Council must be fed research/decisions

## Aliases

- evidence class mismatch
- council abstains on adoption question
- scan is not research

## Status

closed

## Owner

Chief Architect / Orchestrator

## Evidence

- **First real Agent Council run** (`.archetype/council/pydantic-ai-review.json`). The question was an *adoption* decision — "Should ArchetypeOS adopt pydantic-ai as the foundation for its Agent Council's LLM provider abstraction?" — but the only evidence the selectors supplied was a **structural scan of pydantic-ai's own repository** (a bare directory listing reached the architecture agent; the security agent's evidence array was `[]`).
- The Council **correctly abstained** — verdict `Insufficient evidence`, confidence 0.0375 — and the `research_librarian` cited the Engineering Constitution by name ("Research before implementation; Evidence over opinion") and RFC-0004, refusing to recommend. The `follow_up` named exactly what to gather: research notes on pydantic-ai, a technology-fitness comparison (vs. LangChain / instructor / raw SDKs / a custom layer), and a security/dependency-risk review.

## Content

- Event: the Council reasoned over whatever the evidence selectors provided, and for an adoption question that structural scan of the *target* repo was insufficient — so it abstained instead of manufacturing a verdict.
- Source: first real Council run, self-observed.
- Category: design-insight (not a defect — this is the Constitution working as intended).
- Lesson: **match the evidence class to the question class.** A structural scan answers "how is *this* system built?"; an *adoption* question needs a research/decision corpus (fitness comparison, production-readiness notes, security review, prior decisions). Feeding the Council a target-repo scan for an adoption call is a category error; the right fix is upstream — gather and record the research first (RFC-0004 lesson/decision capture), then run the Council over *that*. This is the direct input to the **Phase C decision loop** (Council → gather/record research → draft decision → human-approve → knowledge).
- Loop feed: scoped design input for Phase C. The Council's abstention + `follow_up` is itself the specification for what evidence the decision loop must assemble.
- Status: closed — consumed by **AOS-RESEARCH-001** (RFC-0011 slice-1). The
  Research Engine `research(db, *, project_id, question, ...)` gathers a local
  corpus (repository distillations, prior research notes, decisions — no network),
  ranks it by **source-quality × relevance** (the 7-rung ladder over the calibrated
  need-coverage score, LES-023), synthesizes a deterministic dossier, and persists
  a `ResearchNote` — turning the abstention's `follow_up` ("gather research") into a
  durable, Council-readable evidence class. The close is demonstrated against the
  **real** `synthesize_verdict` (not a staged constant) by
  `test_research_run_clears_council_abstention`: a project with structural evidence
  only (architecture + fitness at deterministic confidence 0.5 each, research class
  empty) abstains — verdict `Insufficient evidence`, aggregate confidence **0.25**
  (mean of `0.0, 0.5, 0.5, 0.0`), `total_evidence 2`; after one `research()` run
  the `research_librarian` agent rises empty→`Complete` (confidence 0.5, evidence 1),
  `total_evidence` rises to 3, the 4-agent mean clears the 0.35 floor at **0.375**,
  and the verdict flips to **`Accept`** — driven solely by the added research-class
  evidence. Calibration finding: because `LocalCorpusSource` reads the same local
  rows the Council's `research_librarian` selector reads, a genuinely research-empty
  baseline yields only an empty note from the default source; the clearing test
  therefore injects a research corpus via the `source=` seam (what `WebResearchSource`
  will supply in slice-2) so the baseline is authentically research-empty while
  `research()` still does real scoring — an honest demonstration of the same loop.
