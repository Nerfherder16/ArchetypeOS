# LES-015 — Playwright dynamic-count assertions must use retrying web-first matchers, not one-shot `await locator.count()`

## Aliases

- e2e count race
- one-shot count vs toHaveCount
- filter-refetch race

## Status

closed

## Owner

## Evidence

- AOS-21: after making `knowledge.spec.ts`'s Open-filter assertion count-agnostic, the Orchestrator's first rewrite read `const openCount = await knowledge.getByRole('listitem').count()` immediately after clicking "Open" and asserted `openCount < allCount`. It failed (`14 < 14` false) because the click triggers an async refetch; a one-shot `.count()` reads the still-unfiltered list before it settles. Playwright's `toHaveCount`/`toBeVisible` are web-first and auto-retry; a bare `await .count()` does not. Caught in the Orchestrator's own full-suite headless run before commit/CI.

## Content

- Event: a count-agnostic e2e assertion raced the filter's async refetch because it used a one-shot `await locator.count()` comparison instead of a retrying assertion.
- Source: AOS-21, self-caught during Orchestrator verification.
- Category: self-found-defect
- Lesson: **for a filtered/refetched list whose exact size is dynamic, assert with retrying web-first matchers, not arithmetic on a one-shot `.count()`.** Prefer proving the filter by *presence/absence of specific rows*: the expected row stays `toBeVisible()`, a known excluded row goes `toHaveCount(0)`. Both retry until the async update settles and stay robust to list growth (composes with LES-012's count-agnostic rule). Reserve `await locator.count()` for values that are stable at read time.
