# LES-044 — PR Guardian's `missing-worker-tests` fires when `apps/worker/app/` changes without an `apps/worker/tests/` change; a handler-path test in the API suite does not count

## Aliases

- missing-worker-tests BLOCK on a new worker handler
- worker code changed without worker test changes
- adding apps/worker/app/handlers/*.py needs an apps/worker/tests/ test
- the enqueue test in apps/api/tests doesn't satisfy the worker-tests gate

## Status

validated

## Owner

Chief Architect / Orchestrator (cloud session)

## Evidence

- PR #222 (AOS-COUNCIL-VALIDATION-API-001) added `apps/worker/app/handlers/foundation_council_review.py`
  + a `registry.py` line, with the handler's behaviour covered only indirectly (the API test asserts
  the route *enqueues* a job; the engine test covers `review_candidate`). PR Guardian BLOCKed:
  "[BLOCK] missing-worker-tests: Worker code changed without worker test changes."
- `tools/pr_guardian.py` `check_tests_for_code_changes`: `worker_changed = any(path.startswith("apps/worker/app/") ...)`
  and `worker_tests_changed = any(path.startswith("apps/worker/tests/") ...)`; `worker_changed and not
  worker_tests_changed` → BLOCK (parallel `api_tests_changed`/`web_tests_changed` gates exist for the
  other apps). The gate is path-based: a test under `apps/api/tests/` does NOT satisfy the *worker* gate,
  even when it exercises the handler transitively.
- Fix: added `apps/worker/tests/test_foundation_council_review_handler.py` — dispatches the handler
  end-to-end via `worker.run_job` (claim → dispatch → complete) with the `DeterministicProvider`, and
  asserts redelivery idempotency (a second `run_job` returns the same review, no duplicate). The worker
  `tests/` dir has no `conftest.py` and is not a package, so the `worker_db` fixture (local to
  `test_worker.py`) is NOT importable via `from .test_worker import worker_db` (relative import with no
  parent package) — define a self-contained `worker_db` fixture in the new module instead.

## Content

- Event: a PR that adds a worker handler (with only an API-side enqueue test + an engine test) BLOCKed
  on `missing-worker-tests` because no file under `apps/worker/tests/` changed.
- Rules:
  1. **Any change under `apps/worker/app/` needs a change under `apps/worker/tests/` in the same diff**
     (the same path-based rule holds for `apps/api/app/`→`apps/api/tests/` and `apps/web/src/`→
     `apps/web/e2e/`). A transitive test in another app's suite does not count — the gate is per-app path.
  2. **Prefer a real handler test** (dispatch through `worker.run_job`, assert completion + persisted
     effect + redelivery idempotency) over a stub — a new handler's claim/dispatch/idempotency path is
     otherwise untested by the engine + API tests.
  3. **The worker `tests/` dir is flat (no `conftest.py`, not a package):** don't `from .test_worker
     import <fixture>` (relative import fails); define the `worker_db`/`FakeRedis` helper locally in the
     new module, or promote it to a `conftest.py` if sharing is worth the broader change.
  4. Add this to the pre-open Guardian checklist for every PR touching a runtime app: capability map
     (LES-040) if a new doc, docs + Acceptance Evidence (LES-041), verification metadata (LES-043), and
     the matching per-app tests (this lesson).
