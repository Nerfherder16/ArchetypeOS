# LES-020 — A core (`aos_core`) change needs a test in the same change set, even a small infra tweak

## Aliases

- missing-core-tests block
- database.py changed without tests
- infra change needs a unit test

## Status

closed

## Owner

Chief Architect / Orchestrator

## Evidence

- AOS-COUNCIL-PHASEC2B: the change added a sqlite WAL/`busy_timeout` `connect` listener to `packages/aos_core/aos_core/database.py` (so the e2e API + worker can share one sqlite file). The PR Guardian returned **BLOCK — `missing-core-tests`: aos_core changed without test changes**, because the only exercise of the new code was the Playwright e2e — which is **not** under the test globs the guardian checks (`packages/aos_core/tests/` or `apps/api/tests/`).
- Root cause: the `missing-core-tests` rule (LES-010 family) treats **any** `aos_core` source change as needing an accompanying test in the same change set — it does not distinguish "small infra tweak" from "new feature," and an e2e that happens to cover the code does not satisfy it.

## Content

- Event: a guardian BLOCK on a frontend-focused PR because its one core-package line (a DB pragma listener) shipped without a unit test.
- Source: PR Guardian, AOS-COUNCIL-PHASEC2B.
- Category: guardian-catch.
- Lesson: **when a package touches `aos_core` at all, add a unit test for the new logic in the same change set** — even a one-function infra change. Extract the decision logic into a testable predicate (here `_is_file_sqlite`) and pin its branches; do not lean on an e2e to "cover" it (the guardian's `missing-core-tests` rule only counts tests under its globs, and rightly so — the e2e is not a substitute for a hermetic unit test of the branch logic). Never override the rule to get green; add the test.
- Loop feed: **consumed in the same change set** — `apps/api/tests/test_database_pragma.py` pins `_is_file_sqlite` (file → WAL on; `:memory:`/empty `sqlite://`/Postgres → untouched).
- Status: closed — cite this row if `missing-core-tests` fires again on an infra-only core change.
