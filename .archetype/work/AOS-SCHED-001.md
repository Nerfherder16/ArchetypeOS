## Design

- **`Schedule` model** (`AuditMixin, Base`): `project_id` (GUID FK nullable), `name` (String), `job_type` (String), `payload` (JSONField default dict), `interval_seconds` (Integer), `enabled` (Boolean default True), `last_run_at` (DateTime tz nullable), `next_run_at` (DateTime tz, default `now_utc` → due immediately on create). Table `schedules`.
- **Migration `0002`**: autogenerate after adding the model (`down_revision = '0001'`) — a single `create_table("schedules")`. The post-migration no-drift autogenerate probe MUST be empty.
- **`aos_core.services.jobs`**: `QUEUE = "archetypeos:jobs"`; `enqueue_job(db, client, *, job_type, project_id=None, repository_id=None, payload=None, priority=100) -> Job` — create the Job (status `queued`), commit, `client.lpush(QUEUE, job.id)` (client is duck-typed; aos_core gains NO redis dependency), return the job. `worker.py` and `main.py` import `QUEUE` from here (single source).
- **`aos_core.services.scheduler`**: `run_due_schedules(db, client, now) -> list[str]` — select enabled `Schedule` where `next_run_at <= now`; for each, `enqueue_job(...)`, set `last_run_at = now`, `next_run_at = now + interval_seconds`; commit; return enqueued job ids. This is the tested tick.
- **`apps/api`**: `POST /jobs` calls `enqueue_job`. New: `POST /projects/{id}/schedules`, `GET /projects/{id}/schedules`, `GET /schedules/{id}`, `PATCH /schedules/{id}` (enabled/interval/name), `DELETE /schedules/{id}`, `POST /schedules/{id}/run` (enqueue immediately via enqueue_job, set last_run_at). 404 conventions as established.
- **`apps/scheduler`**: thin. `app/main.py` = loop: `client = redis.from_url(settings.redis_url)`; every ~30s `with SessionLocal() as db: run_due_schedules(db, client, now_utc())`; sleep. `requirements.txt` = redis (aos_core provides the rest). Dockerfile repo-root context (COPY packages/aos_core + install, COPY apps/scheduler/app). compose `scheduler` service (single instance, depends_on postgres+redis, same env as worker).
- **Scheduler logic lives in aos_core** (guarded + tested); `apps/scheduler/app/main.py` is a trivial entrypoint, so no guardian change is needed this package (extending guardian test-enforcement to `apps/scheduler/` is a noted follow-up).

## Acceptance Criteria

- Schedule is a durable row — evidence: `Schedule` model + migration `0002`; `test_schedule_crud` (create/list/read/patch/delete via API).
- No schema drift after the new migration — evidence: `alembic upgrade head` → 22 tables incl. `schedules`; autogenerate no-drift probe → 0 ops.
- Scheduler materializes due schedules into jobs — evidence: `test_run_due_schedules` — a due schedule → a Job row (queued) + a fake-redis `lpush` + `last_run_at`/`next_run_at` advanced; a not-yet-due schedule → nothing.
- One enqueue path — evidence: `POST /jobs` and the scheduler both go through `enqueue_job`; `QUEUE` imported from `aos_core.services.jobs` in worker + api.
- Run-now works — evidence: `test_schedule_run_now` (`POST /schedules/{id}/run` → job enqueued).
- Nothing broken — evidence: api suite (69 + new) green; worker suite (5) green; ruff/compile clean; alembic no-drift.
- Postgres + new service — evidence: CI compose-smoke green (migration 0002 applies on fresh Postgres; the scheduler container builds + boots).
