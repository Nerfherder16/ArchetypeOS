## Evidence

Each P0/P1 claim in the July-10 2026 evaluation was checked by reading the cited code on `main` (adversarial: agents were told to actively look for refuting evidence — a unique constraint, a lease column, a reconciliation job, an auth dependency, an enforcement call site). All eleven confirmed. Load-bearing citations:

- **P0-1 (non-atomic job enqueue, no lease) — CONFIRMED.** `packages/aos_core/aos_core/services/jobs.py:36-38`: `db.commit()` → `db.refresh(job)` → `client.lpush(QUEUE, job.id)` with no try/except and no compensation; a Redis failure after the commit orphans a `queued` row. Worker `apps/worker/app/worker.py:167` uses `brpop` (removes the id before `run_job` at :173); a SIGKILL/OOM between pop and completion loses the job — the `except` never runs. `Job` model (`models.py:288-301` + `AuditMixin` :48-56) has no `claimed_by`/`lease_expires_at`/`idempotency_key`. No outbox table across all 15 Alembic migrations; no reconciliation scanner. Only mitigation is `handle_failure` (`worker.py:150-160`, `MAX_ATTEMPTS=3`) which fires solely on a *catchable* exception; no worker heartbeat; no dead-letter queue (terminal failure just flips `status="failed"`).
- **P0-2 (scheduler duplicates jobs) — CONFIRMED.** `packages/aos_core/aos_core/services/scheduler.py:18-38`: `enqueue_job` self-commits each job; `next_run_at`/`last_run_at` advance in a single `db.commit()` after the loop → a crash between re-fires next tick. Plain `SELECT` (no `with_for_update(skip_locked=True)`, no `pg_advisory_lock`, no leader election) — two replicas double-enqueue. No `ScheduleFire` entity / `(schedule_id, nominal_fire_at)` uniqueness. `next_run_at = now + interval` (:36) coalesces missed runs, never backfills.
- **P0-3 (handler retries duplicate artifacts) — CONFIRMED.** `apps/worker/app/worker.py:146-147`: `spec.run(job, db)` commits domain output, then `mark_job(job_id,"completed")` commits in its own separate `SessionLocal` (`worker.py:30,43`). Council review commits at `:91-92`, digest `:74-77`, research note `research.py:482-483`, research run `research_run.py:131`. `CouncilReview.job_id`/`ResearchRun.job_id`/`Artifact.job_id` are `index=True` but **not unique**; `NightlyDigest`/`ResearchNote` carry no `job_id`. A retry re-runs the handler → duplicate artifacts. (`AuditHeartbeat` proves the codebase knows how to add a `UniqueConstraint`; it just isn't applied to job outputs.)
- **P0-4 (connector config truth split API↔worker) — CONFIRMED.** Catalog marks `free_llm_pool` configured via `bool(s.llm_free_api_key)` (`connectors.py:77`), but the real rotating pool keys on `GROQ_API_KEY`/`CEREBRAS_API_KEY`/`GEMINI_API_KEY`/`MISTRAL_API_KEY` from `os.environ` (`llm_pool.py:38-49,112-116`), and `docker-compose.yml:94-97` forwards those four keys to the **worker only** (the `api:` block never receives them). `local_llm` is "configured" from a non-empty `llm_base_url` default `http://localhost:11434/v1` (`config.py:29`) with no reachability probe. `GET /connectors` calls `sync_connectors` which INSERT/UPDATE + `db.commit()` on every read (`connectors.py:178`) — a mutating GET, further bounded by an `@lru_cache` settings snapshot captured at import (`connectors.py:22`).
- **P0-5 (node/connector mutation has no identity) — CONFIRMED.** Every route in `apps/api/app/routes/nodes.py:19-33` has only `Depends(get_db)` — no token, no auth; `register()` passes attacker-controlled `write_access`/`max_sensitivity`/`capabilities` into `register_node`, which overwrites any node *by name*. `POST /connectors/{name}/health` is identically unauthenticated. Only middleware in `main.py` is CORS. `audits.py:22` *does* use an `x_telemetry_token` header — the pattern exists but is not applied here.
- **P0-6 (authority is advisory, not enforced) — CONFIRMED.** `requires_approval()` (`services/authority.py:89-102`) is a pure function; the only callers are the service itself, three informational routes (`routes/authority.py`), and tests/docs. `jobs.py`, `worker.py`, and the research/distillation/repositories/nodes/connectors services contain **zero** authority calls (grep of `worker.py` for `authority|requires_approval|ActionClass` → no matches). A caller can `POST /jobs` and the worker executes with no authority consultation; the engine's "cannot be silently bypassed" docstring is aspirational.
- **P1-1 (handler registry still centralized) — CONFIRMED.** if/elif replaced by a `JOB_HANDLERS` dict, but all imports/functions/registrations remain inline in `worker.py` (registration block `:121-126`); `HandlerSpec` (`:46-58`) declares only `job_type`/`capability`/`sensitivity`/`run` — no timeout/retry/idempotency/result-schema; `sensitivity` is read only by a test assertion, never enforced; no `apps/worker/app/handlers/` package exists.
- **P1-2 (node registry not connected to execution) — CONFIRMED.** The worker never calls `register_node`/`record_heartbeat`, never claims by capability (blind `brpop` on one queue), never checks `max_sensitivity`/`write_access`, never reads `Node.endpoint`. No `aos-node` daemon exists anywhere. The registry is CRUD consumed only by the API/dashboard.
- **P1-3 (node/heartbeat constraints not concurrency-safe) — CONFIRMED.** `register_node` query-then-insert with `Node.name` non-unique (`models.py:519`; migration 0010 has no `unique=True`); no `(node_id, capability)` uniqueness (`NodeCapability` `models.py:533-542`); `AuditHeartbeat` `UniqueConstraint("routine","project_id")` with nullable `project_id` (`models.py:222-232`, migration 0015) → Postgres treats NULLs as distinct, so concurrent global `(routine, NULL)` inserts duplicate; service does query-then-insert (`audit_heartbeat.py:34-41`).

Staleness is semantic (human narrative), not the auto-derived watermark.
- **P1-5 (web deploy is a dev server) — CONFIRMED.** `docker-compose.yml:136-151` runs `apps/web/Dockerfile` whose `CMD ["npm","run","dev"]` (`"dev": "vite --host 0.0.0.0"`) serves the Vite dev server on 5173; no multi-stage build, no nginx/caddy, no `/api` proxy (browser hits `VITE_API_BASE_URL` directly). ## Content

- **Event:** an operator-supplied whole-repository evaluation (July-10 2026) raised six P0 and five P1 architectural findings. Rather than acting on them by inference, each claim was adversarially verified against the code on `main`. All eleven confirmed with file:line evidence (above). No refuting constraint, lease, outbox, auth dependency, or enforcement call site was found for any claim.
- **Source:** operator-directed evaluation + in-session adversarial code verification, 2026-07-10/11. Verification only — no code changed.
- **Category:** self-found-defect (a review-class event: the system's own runtime-integrity defects, confirmed against source).
- **Lesson (transferable):**
  1. **An entity that exists as data or as a pure function is not an enforcement boundary.** The Node registry, Connector catalog, and Authority evaluator are all *implemented* — models, routes, a `requires_approval()` — yet none is on any execution path: the worker never consults nodes, egress never consults connector policy, and no write/deploy/destructive path calls authority. "It exists in code" ≠ "it governs execution." When a design promises a boundary (the authority docstring literally says it "cannot be silently bypassed"), verify a *call site* on the path it claims to guard, not just the definition.
  2. **Durability defects are invisible to tests that assert selection and shape but never drive the crash path.** Router/selection tests passed while `route()` ignored the offline signal (LES-L19); assert-transparency tests passed while a sink recorded zero events (LES-032); here, unit tests assert `HandlerSpec.sensitivity in {"public","private"}` and that handlers return a dict, but nothing kills the worker between `brpop` and completion, or between the domain commit and the job commit. To catch at-least-once/exactly-once gaps you must test the failure: crash after commit-before-push, redeliver a job, run two schedulers, insert two `(routine, NULL)` heartbeats concurrently.
  3. **A default that makes a tier/connector "available" is not proof it is reachable or that the same process owns it** (extends LES-L19/L20). `llm_base_url=localhost` reads as "configured" with nothing listening; the free pool's real keys live in the worker while the catalog that reports "configured" runs in the API. Split the connector truth into declared / credential-present / reachable / healthy, and load it identically in every process.
  4. **A machine-refreshed watermark can be current while the human narrative rots.** P1-4's watermark is accurate (`#183`); the Objective and Open-Decisions fields are stale. Auto-derivation solved branch-name/PR-lag drift (LES-L09), not truth drift. Derive execution state from structured records; don't let a green watermark certify hand-written prose that still calls shipped systems "Proposed".
