# Repository Intelligence Engine

## Purpose

The Repository Intelligence Engine teaches ArchetypeOS how to study repositories deeply.

It treats a repository as an engineering artifact to reverse engineer, evaluate, mine for patterns, and convert into durable portfolio knowledge.

## Core Principle

Every repository reviewed by ArchetypeOS should permanently increase the engineering intelligence of the portfolio.

## Mission

Do not merely summarize a README. Inspect the full repository structure, code, runtime, docs, workflows, configuration, tests, issues where available, and architectural patterns.

## Phase 1: Discovery

Capture basic metadata:

- repository purpose
- maturity
- license
- last activity
- release cadence
- contributors
- language mix
- dependencies
- package managers
- CI/CD
- Docker and deployment files
- infrastructure files
- database usage
- AI providers
- APIs
- MCP usage
- browser or desktop automation
- local runtime assumptions

## Phase 2: Architecture Reverse Engineering

Identify:

- services
- modules
- layers
- interfaces
- boundaries
- trust zones
- dependency graph
- runtime topology
- deployment topology
- authentication
- data flow
- event flow
- message queues
- background workers
- plugin systems
- agent orchestration

Output an editable Architecture Spine Graph draft.

## Phase 3: Pattern Mining

Find reusable patterns:

- authentication
- provider abstraction
- plugin system
- agent system
- workflow engine
- prompt management
- vector search
- caching
- retry logic
- configuration
- feature flags
- permissions
- authority gating
- multi-tenancy
- scheduling
- desktop automation
- browser automation
- wake word handling
- global hotkeys

Each pattern should receive:

- maturity
- complexity
- reuse score
- portability
- dependencies
- risks
- recommended adaptation path

## Phase 4: Quality Audit

Score the repository across:

- Architecture
- Security
- Documentation
- Maintainability
- Testing
- Scalability
- Innovation
- Reuse Potential
- AI Readiness
- Portfolio Fit

## Phase 5: Knowledge Extraction

Generate durable artifacts:

- ADR candidates
- lessons learned
- architecture decisions
- risks
- benchmarks
- experiments
- anti-patterns
- best practices
- knowledge packs
- recommendation cards

## Phase 6: Reuse Analysis

Identify what ArchetypeOS can use.

For each reusable item, include:

- name
- source files
- conceptual summary
- why it matters
- reuse score
- adaptation effort
- risk
- license concerns
- estimated engineering savings

## Phase 7: Portfolio Impact

Compare the reviewed repository against the current portfolio.

Questions:

- What does this repo solve that we need?
- Which current projects could reuse its patterns?
- Which planned ArchetypeOS engines are strengthened by it?
- What should be imported as a concept only?
- What should be avoided?

## Repository Research Rubric

Every deep repository review should include:

- Executive Summary
- Architecture
- Runtime
- Infrastructure
- AI Stack
- Data Model
- UX
- Workflow
- Security
- Compliance
- Performance
- Innovation
- Reusability
- Weaknesses
- Lessons Learned
- Patterns Worth Adapting
- Patterns To Avoid
- Overall Rating
- Recommended Imports
- Estimated Engineering Savings

## Supported Research Targets

The same rubric can extend beyond GitHub repositories:

- MCP servers
- VS Code extensions
- Claude skills
- Docker images
- APIs
- SDKs
- SaaS products
- UI libraries
- Figma kits
- academic papers
- engineering blogs
- architecture talks
- conference presentations

## Safety And Licensing

Do not copy code until license compatibility is verified.

Prefer importing architecture patterns, interfaces, and concepts first.

## Principle

Repository review is not one-off research. It is knowledge acquisition for the entire engineering portfolio.
