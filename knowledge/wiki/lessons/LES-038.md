- **The machine watermark was current while the human narrative rotted — again.** `CURRENT_STATE.md`'s auto-derived fields showed Watermark PR `#204`, but the human-authored Open Decisions table still called three now-ENFORCED subsystems "Implemented (registry); routing pending", "Implemented (registry); runtime unification pending", and "Implemented (advisory); enforcement pending". The detector reported FRESH on a doc that was lying.

## Content

- **Event:** the wave shipped enforcement (registries/evaluators moved onto the execution path) but the canonical state docs still described those subsystems as advisory/pending, and the drift detector meant to catch exactly this (P1-4) was blind to it because it keyed on file existence and a narrow token set, and even a true positive would only WARN.
- **Category:** self-found-defect (found while reconciling state, verified against the detector's own logic).
- **Lesson (transferable):**
  1. **A "does the subsystem exist" marker must be an ENFORCEMENT symbol, not a file path.** A long-lived module file is present in both the advisory and the enforced era, so its existence proves nothing about whether the label is honest. Key the marker on a symbol that only exists once the behaviour is on the execution path (`claim_job_for_node`, `consume_action`, `connector_views`) — "file exists AND contains the symbol" is the signal that the enforcement, not merely the module, shipped.
  2. **Contradiction tokens must track how the label actually rots over time, not one snapshot.** The first pass hunted "Proposed"; the wave produced "advisory"/"pending"/"routing pending". Carry a generic contradiction set plus per-subsystem stale phrasings, and widen it whenever a new stale wording is observed — the token list is itself a living part of the detector.
  3. **A clear contradiction should BLOCK; only a lag should WARN.** A normal one-PR reconciliation lag (watermark/roadmap) is expected and stays advisory. But a state doc asserting a subsystem is advisory/pending when its enforcement symbol has shipped is an unambiguous lie, not a lag — promote `semantic-label-stale` to a guardian BLOCK while keeping the detector fail-open (a missing/erroring detector yields no findings, so it can never itself wedge a merge).
  4. **Keep the detector a pure function of an injected marker set.** `check_semantic_labels(text, present_markers)` takes the marker set as an argument so the enforced/advisory/absent cases are all fixture-testable without touching disk; a thin `_present_markers(repo_root)` does the one impure read at the CLI edge.
- **Loop feed:** the enforcement-symbol marker pattern generalizes to any future "is this really on the execution path" claim (council/research egress, remote node execution) — add a marker when the next subsystem lands; the block-on-contradiction rule makes the canonical docs self-defending against the "watermark current, narrative stale" failure mode rather than merely observing it.
- **Status:** closed — consumed by AOS-STATE-RECON-002 (branch `claude/aos-state-recon-002`): symbol-keyed `check_semantic_labels` + `_present_markers`, broadened contradiction tokens, `semantic-label-stale` → guardian BLOCK (fail-open preserved), and the CURRENT_STATE/RECENT_CHANGES/HANDOFF reconciliation to the merged wave (detector reports FRESH).
