# LES-040 — an RFC / docs-only PR that introduces a *planned* capability still trips `capability-map-not-updated`; register the proposed capability in its layer file, don't override

## Aliases

- Guardian BLOCK `capability-map-not-updated` on a docs-only / RFC PR
- new RFC added but PR Guardian BLOCKs on the capability map
- do I override or update the capability map for an RFC that ships no code
- where does a proposed (not-yet-built) capability go in the capability map

## Status

validated

## Owner

Opus orchestrator (cloud session)

## Evidence

- PR #208 added three docs (`docs/rfc/RFC-0016-…`, `docs/FOUNDATION_INTELLIGENCE_DESIGN_V0_1.md`,
  `knowledge/wiki/reviews/2026-07-12-…`) and shipped **no code**. All CI jobs were green
  except **PR Guardian**, which BLOCKed in ~8s (deterministic finding, not a test run):
  `[BLOCK] capability-map-not-updated: New docs were added without updating the capability
  map (docs/CAPABILITY_MAP.md or docs/capability-map/layer-NN.md). Add map changes or include
  PR_GUARDIAN_OVERRIDE_CAPABILITY_MAP with rationale.`
- Distinct from [[LES-L14]]: there the same finding was a **diff-base** artifact (a stray
  ancestor commit's files). Here the finding is **correct** — new docs really were added
  and the map really was untouched. The question is which of the two offered resolutions is
  aligned, not how to make the finding go away.
- Resolution chosen: **update the map, not override.** The capability map (`docs/CAPABILITY_MAP.md`
  → `docs/capability-map/layer-NN.md`) tracks how capabilities fit into the platform's
  *plan* by layer, not only shipped code, so a governance artifact that introduces a new
  capability family belongs in it. Foundation Intelligence is fundamentally an evidence
  model → **Layer 2 (Research and Evidence)**; registered there as `(proposed, RFC-0016)`
  with the design doc + RFC as primary artifacts, cross-referencing Layers 3 (genome) and 4
  (foundation candidates/baselines). That cleared the BLOCK legitimately.

## Content

- Event: a docs-only RFC PR was BLOCKed by `capability-map-not-updated`; the finding was
  correct (new docs, untouched map), so the fix is content, not history.
- Root cause (by design): PR Guardian treats *any* new doc under tracked doc trees as a
  potential capability addition and requires the map to move with it, forcing the map to
  stay a true index of the platform rather than drifting behind a pile of docs.
- Decision rule — override vs. update:
  1. If the PR introduces or changes a **capability** (even a *proposed*, not-yet-built one
     via an RFC), **update the map** — add the capability to the correct `layer-NN.md` with a
     `(proposed, RFC-NNNN)` marker and list the RFC/design doc as primary artifacts. This is
     the aligned fix; the map indexes the plan, not just merged code.
  2. Reserve `PR_GUARDIAN_OVERRIDE_CAPABILITY_MAP: <rationale>` for docs that genuinely add
     no capability (e.g. a review note, a lesson, a runbook edit) — and even then prefer
     letting the map's existing artifact lists absorb the reference.
  3. Pick the capability's **home layer by its essence**, not by which doc dir it lives in:
     an evidence/claims capability → Layer 2; architecture/model → Layer 3; decision/
     recommendation → Layer 4. Cross-reference the other layers it touches in one line
     rather than duplicating the entry across files (the map is split per layer for
     token-frugal reads + conflict isolation, LES-L07).
  4. A fast (~seconds) Guardian job with `Verdict: BLOCK` and one finding is deterministic —
     read the single finding and fix exactly it; don't re-run hoping it flips.
