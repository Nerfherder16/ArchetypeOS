## Content

- Event: closing LES-007 (a correct, in-scope consumption record for AOS-20) broke two test suites that pinned it as the canonical OPEN lesson — one in the api layer, one in the web e2e layer.
- Source: PR #68 CI (self-found on the first CI run of AOS-20).
- Category: ci-failure.
- Lesson: tests over the lessons vault must derive the open set live (`OPEN_IDS` parsed from the index; the `(lesson · open)` badge generically), never assert a specific lesson ID is perpetually open. A lesson's status flips the moment a loop consumes it — that is the whole point of the learning loop — so an open-status assertion pinned to one ID is guaranteed to rot. Closed lessons are stable anchors; open ones are not. This is the same class as LES-006/LES-012/LES-015 (do not couple tests to mutable vault specifics).
- Loop feed: fixed in the same change set. `test_knowledge.py` now asserts the parsed/synced open set equals the live `OPEN_IDS` and picks any open page for the fetch check; `knowledge.spec.ts` asserts the `(lesson · open)` badge generically and uses "Doc staleness" (LES-007) only as a stable CLOSED anchor.
- Status: closed.
