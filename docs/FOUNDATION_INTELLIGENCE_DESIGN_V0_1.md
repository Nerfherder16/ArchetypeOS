# Foundation Intelligence — Conceptual Design v0.1

> **Source of record.** This is the operator-supplied conceptual design for the
> ArchetypeOS *Project Foundation Intelligence* capability (System Genome, Evidence
> Model, and Foundation-Selection Lifecycle), filed verbatim as the durable "source"
> node of the §21 traceability chain. `docs/rfc/RFC-0016` is the governing RFC that
> adjudicates it; `knowledge/wiki/reviews/2026-07-12-foundation-intelligence-design-review.md`
> is the senior review that produced RFC-0016. Do not edit this file to reflect later
> decisions — decisions live in the RFC. Corrections to the *design intent* create a
> v0.2 alongside this file.

---

## 1. Purpose

ArchetypeOS must be able to ingest an incomplete, contradictory collection of
repositories, documents, communications, research, simulations, requirements, and prior
decisions and determine:

1. What system currently exists.
2. What system stakeholders intended to construct.
3. What system should be constructed.
4. Which engineering foundations best fit that target.
5. Which assumptions and risks must be validated before implementation.
6. How every conclusion can be traced back to evidence.

The output is not merely an architecture diagram or technology recommendation. It is a
versioned, evidence-backed foundation baseline from which implementation plans, agent
work packages, acceptance criteria, and verification gates can be generated.

ArchetypeOS already models repositories, Repository DNA, architecture nodes and edges,
decisions, research, recommendations, and implementation plans. The models introduced
here should connect those capabilities rather than replace them.

## 2. Core Domain Separation

The system must keep five concepts distinct.

| Concept | Meaning |
|---|---|
| Repository DNA | What one repository reveals about its implementation, frameworks, services, capabilities, maturity, and risks |
| Evidence Graph | What the project corpus says, proves, implies, disputes, or decides |
| System Genome | What kind of complete engineered system the evidence describes |
| Foundation Candidate | One coherent way of building or evolving that system |
| Foundation Baseline | The approved, versioned foundation governing future implementation |

The most important boundary is:

> Repository DNA describes code. The System Genome describes the engineered system
> surrounding the code.

A project may contain several repositories, hardware, human workflows, vendor
dependencies, legal constraints, simulations, and operating procedures. No individual
repository can describe the whole system.

## 3. Shared Truth Model

Every material statement in ArchetypeOS must belong to one of four truth layers.

| Truth layer | Definition | Examples |
|---|---|---|
| `observed` | Directly established through deterministic inspection, execution, measurement, or authenticated records | A dependency exists in a lockfile; a test failed; a service listens on a port |
| `claimed` | Asserted by a person, document, communication, external source, or prior agent | "The system must operate offline"; "the customer expects 10,000 users" |
| `inferred` | Derived by ArchetypeOS from one or more observations or claims | The system appears latency-sensitive; a repository likely implements a control plane |
| `decided` | Formally approved by an authorized person or policy process | PostgreSQL selected; cloud egress prohibited; modular monolith approved |

These layers must never be silently merged. A statement does not become `observed`
merely because an LLM generated it. A claim does not become a decision because it appears
in a design document. A decision does not prove that implementation conforms to it.

## 4. Evidence Model

### 4.1 Evidence architecture

The evidence model should be claim-centric rather than document-centric. A document is a
source container. The useful reasoning unit is a specific claim extracted from a specific
location in a specific version of that source.

```
EvidenceSource
    └── EvidenceSourceVersion
            └── EvidenceFragment
                    └── Claim
                          ├── supporting evidence
                          ├── opposing evidence
                          ├── related claims
                          ├── genome traits
                          ├── architecture elements
                          ├── risks
                          └── decisions
```

### 4.2 EvidenceSource

Represents the logical source. Examples: Git repository, uploaded document, email thread,
legal communication, simulation package, test run, meeting transcript, external
specification, research paper, API response, production log, human interview, image or
diagram.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "source_type": "repository|document|communication|simulation|test_run|runtime_record|external_reference|human_input|diagram|dataset",
  "title": "string",
  "origin": "github|upload|email|local_filesystem|web|connector|manual|generated",
  "originator": "person, organization, system, agent, or repository",
  "canonical_uri": "optional source URI",
  "sensitivity": "public|internal|confidential|restricted|privileged",
  "authority_domains": ["runtime", "product", "legal", "security"],
  "access_policy_id": "optional uuid",
  "status": "active|superseded|withdrawn|unavailable",
  "created_at": "timestamp"
}
```

### 4.3 EvidenceSourceVersion

Sources must be versioned because repositories, contracts, requirements, and simulations
change.

```json
{
  "id": "uuid",
  "source_id": "uuid",
  "version_ref": "commit SHA, file hash, message id, revision, or timestamp",
  "content_hash": "sha256",
  "captured_at": "timestamp",
  "effective_from": "optional timestamp",
  "effective_until": "optional timestamp",
  "supersedes_version_id": "optional uuid",
  "ingestion_method": "connector|upload|scan|manual|generated",
  "parser_version": "string"
}
```

Evidence versions should be immutable. Corrections create new versions.

### 4.4 EvidenceFragment

A fragment identifies the exact portion of a source supporting a claim.

```json
{
  "id": "uuid",
  "source_version_id": "uuid",
  "locator": {
    "path": "optional repository or document path",
    "start_line": 120,
    "end_line": 138,
    "page": 7,
    "section": "Security Requirements",
    "timestamp_start": "optional",
    "timestamp_end": "optional",
    "json_pointer": "optional"
  },
  "content_hash": "sha256",
  "excerpt": "bounded extracted text or structured value",
  "extraction_method": "deterministic|parser|agent|human",
  "extraction_confidence": 0.96
}
```

The locator structure should permit code lines, PDF pages, spreadsheet ranges, video
timestamps, JSON paths, email message IDs, and simulation result identifiers.

### 4.5 Claim

A Claim is the central reasoning primitive.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "statement": "The deployment environment must not require public cloud connectivity.",
  "claim_type": "fact|requirement|constraint|preference|hypothesis|finding|risk|assumption|decision_candidate|definition",
  "truth_layer": "observed|claimed|inferred|decided",
  "domain": "deployment",
  "scope": {
    "system_id": "optional",
    "repository_ids": [],
    "component_ids": [],
    "environment": "production",
    "applicable_version": "optional"
  },
  "polarity": "affirming|negating|qualifying",
  "confidence": 0.91,
  "materiality": "informational|low|medium|high|critical",
  "status": "active|disputed|superseded|rejected|resolved",
  "valid_from": "optional timestamp",
  "valid_until": "optional timestamp",
  "created_by": "agent, person, or deterministic tool",
  "derivation": {
    "method": "direct|extracted|aggregated|inferred|approved",
    "parent_claim_ids": []
  }
}
```

Claim classifications (initial controlled vocabulary):

| Category | Meaning |
|---|---|
| `fact` | Description of something believed to be true |
| `requirement` | Capability or quality the system must provide |
| `constraint` | Boundary the solution may not violate |
| `preference` | Desired but nonbinding characteristic |
| `hypothesis` | Testable proposition |
| `finding` | Result of research, inspection, or evaluation |
| `risk` | Potential adverse outcome |
| `assumption` | Unverified proposition currently relied upon |
| `decision_candidate` | Proposed choice awaiting authorization |
| `definition` | Agreed terminology or boundary |

### 4.6 ClaimEvidenceLink

Evidence does not simply attach to a claim. It has a relationship to the claim.

```json
{
  "claim_id": "uuid",
  "fragment_id": "uuid",
  "relationship": "supports|opposes|qualifies|originates|verifies|invalidates",
  "relevance": 0.95,
  "strength": "weak|moderate|strong|direct",
  "notes": "string"
}
```

### 4.7 Authority, confidence, relevance, and freshness

These must be independent.

- **Authority:** Is this source entitled to define this subject?
- **Confidence:** How likely is the extraction or inference to be correct?
- **Relevance:** How directly does the evidence apply to this claim?
- **Freshness:** Is it still applicable to the current project state?

Authority must be domain-specific. A contract may have high authority for legal
constraints but no authority over runtime behavior. A repository has high authority for
current implementation but may have low authority for business intent. A stakeholder
interview may have high authority over product goals but not security controls. There
should be no universal source ranking that treats one source as superior in all contexts.

### 4.8 Claim relationships

Claims form a graph. Supported initial relationships: `supports`, `contradicts`,
`supersedes`, `refines`, `duplicates`, `depends_on`, `derived_from`, `implements`,
`violates`, `applies_to`, `validated_by`, `invalidated_by`.

### 4.9 Conflict model

A contradiction must remain visible until explicitly resolved.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "claim_ids": ["uuid-a", "uuid-b"],
  "conflict_type": "direct_contradiction|scope_mismatch|temporal_conflict|authority_conflict|implementation_drift|ambiguity",
  "materiality": "low|medium|high|critical",
  "status": "open|accepted_exception|resolved|superseded",
  "resolution": "optional explanation",
  "resolution_decision_id": "optional uuid",
  "blocking_stages": ["foundation_selection", "baseline_approval"]
}
```

Critical unresolved conflicts should block foundation approval when they materially affect
architecture, legality, safety, security, cost, or feasibility.

### 4.10 Sensitivity and compartmentalization

Sensitive sources must not automatically become available to every agent. Required
controls: source-level sensitivity; claim-level sensitivity inheritance; agent capability
restrictions; provider egress restrictions; privileged/legal compartment; redacted
derivative claims; complete access audit trail.

A legal source may produce a redacted claim such as: "The system may not retain category X
beyond 30 days." Most agents can use that claim without receiving the underlying
privileged correspondence.

## 5. Corpus Snapshot

All major analyses must execute against a frozen corpus snapshot.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "source_version_ids": [],
  "repository_refs": [
    { "repository_id": "uuid", "commit_sha": "abc123", "branch": "main" }
  ],
  "claim_set_hash": "sha256",
  "created_at": "timestamp",
  "created_by": "system",
  "purpose": "genome_generation|foundation_selection|drift_review"
}
```

This prevents a Genome or foundation recommendation from silently changing because a
source changed midway through analysis.

## 6. System Genome Schema

### 6.1 Definition

The System Genome is a versioned, multidimensional classification of the entire engineered
system. It must answer: What is being built? Where does it run? What interacts with it?
What does it control? What data does it handle? How autonomous is it? How critical is
failure? Which constraints govern its design? What lifecycle and organizational conditions
shape implementation?

### 6.2 State views

Every Genome Snapshot applies to one state view.

| State view | Definition |
|---|---|
| `current` | What demonstrably exists |
| `intended` | What the corpus indicates stakeholders sought to construct |
| `target` | The approved or proposed future system |
| `candidate` | A temporary classification associated with one foundation option |

A project can have multiple snapshots over time, but only one active snapshot per state
view and baseline version.

### 6.3 GenomeSnapshot

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "corpus_snapshot_id": "uuid",
  "state_view": "current|intended|target|candidate",
  "version": 1,
  "status": "draft|reviewed|approved|superseded",
  "summary": "string",
  "primary_archetypes": [],
  "secondary_archetypes": [],
  "coverage": 0.84,
  "aggregate_confidence": 0.79,
  "open_question_count": 6,
  "critical_conflict_count": 1,
  "generated_by_run_id": "uuid",
  "approved_by": "optional",
  "approved_at": "optional timestamp"
}
```

### 6.4 GenomeTrait

A Genome is composed of individual evidence-backed traits.

```json
{
  "id": "uuid",
  "genome_snapshot_id": "uuid",
  "dimension": "runtime_topology",
  "trait_key": "local_first",
  "value": true,
  "value_type": "boolean",
  "classification": "primary|secondary|conditional|absent|unknown",
  "confidence": 0.93,
  "stability": "stable|evolving|volatile|unknown",
  "criticality": "informational|important|foundation_shaping",
  "rationale": "The system is intended to operate from privately managed infrastructure and local execution nodes.",
  "supporting_claim_ids": [],
  "opposing_claim_ids": [],
  "source_methods": ["repository_scan", "document_analysis"],
  "human_locked": false
}
```

### 6.5 Initial Genome dimensions

- **A. Mission and product form** — primary purpose; user/operator type; business
  capability; product vs internal platform; research/commercial/operational/regulatory purpose.
- **B. Domain** — developer tooling, fintech, healthcare, industrial, embedded,
  infrastructure, consumer, enterprise workflow, scientific, legal, education, other. A
  project may have several domains.
- **C. Lifecycle and maturity** — idea, research, PoC, prototype, pilot, pre-production,
  production, legacy, migration, recovery, maintenance, decommissioning.
- **D. System composition** — software-only; hardware+software; human workflow+software;
  multi-repository platform; multi-organization system; cyber-physical; data product;
  model-centric system.
- **E. Runtime topology** — monolithic; modular monolith; service-oriented; microservices;
  event-driven; distributed workers; peer-to-peer; edge; embedded; batch pipeline;
  interactive application; control plane with execution nodes.
- **F. Deployment and ownership** — local-first; self-hosted; managed cloud; hybrid;
  air-gapped; edge deployed; customer-hosted; multi-tenant; single-tenant; offline-capable.
- **G. Workload characteristics** — interactive; real-time; near-real-time; batch;
  long-running workflow; bursty; compute-heavy; storage-heavy; network-sensitive;
  latency-sensitive; deterministic; probabilistic.
- **H. Data profile** — structured; unstructured; graph; time-series; vector; media;
  source code; personal data; financial data; health data; confidential business data;
  legal/privileged data; safety-relevant telemetry.
- **I. Integration profile** — external API dependent; vendor platform dependent; hardware
  protocol dependent; repository connector dependent; human approval dependent;
  event/webhook driven; intermittent-connectivity tolerant.
- **J. AI and autonomy** — no AI; AI-assisted; AI-generated recommendations; agentic
  execution; human-in-the-loop; human-on-the-loop; autonomous; model-provider agnostic;
  bring-your-own-model; local inference required; explainability required.
- **K. Assurance and criticality** — experimental; low consequence; commercial production;
  business critical; mission critical; safety relevant; safety critical; financially
  regulated; legally sensitive; auditable; high availability required.
- **L. Security and privacy** — public; internal; confidential; restricted; tenant
  isolated; zero-trust boundary; secret-bearing; sensitive egress restricted; data
  residency constrained; strong identity required.
- **M. Reliability and continuity** — disposable; recoverable; durable workflow required;
  idempotent processing required; offline continuity required; disaster recovery required;
  graceful degradation required; exactly-once effect semantics required.
- **N. Organizational context** — solo developer; small team; distributed team;
  enterprise; contractor-led; open source; mixed human/agent team; limited operational
  capacity; regulated governance process.
- **O. Economic and delivery constraints** — fixed budget; fixed deadline; low operational
  cost; minimal vendor lock-in; rapid validation; long-term maintainability; existing
  infrastructure reuse; licensing constraints; limited specialist availability.
- **P. Change profile** — greenfield; brownfield; incremental extension; replatforming;
  rewrite; migration; compatibility constrained; high reversibility required; continuous
  evolution.

### 6.6 Primary system archetypes

Traits should produce a small number of readable archetypes.

```json
{
  "primary_archetypes": [
    { "name": "Local-First Engineering Control Plane", "confidence": 0.94, "trait_ids": [] },
    { "name": "Human-Governed Multi-Agent Execution Platform", "confidence": 0.91, "trait_ids": [] }
  ],
  "secondary_archetypes": [
    { "name": "Repository Intelligence and Institutional Memory System", "confidence": 0.87, "trait_ids": [] }
  ]
}
```

These names are summaries, not substitutes for the underlying traits.

### 6.7 Genome quality indicators

Every Genome Snapshot must expose: evidence coverage by dimension; confidence by trait;
number of unsupported traits; number of opposing claims; number of unresolved material
conflicts; number of open foundation-shaping questions; dimensions marked unknown; human
corrections; difference from the previous snapshot. An apparently confident Genome with
weak evidence must not appear complete.

## 7. Open Questions Model

ArchetypeOS should not ask broad generic interviews. It should create targeted questions
only where answers could alter foundation selection.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "genome_snapshot_id": "uuid",
  "question": "Must the system continue operating when the control plane is unreachable?",
  "affected_dimensions": ["runtime_topology", "reliability"],
  "affected_foundation_domains": ["distributed_runtime", "data_consistency"],
  "materiality": "high",
  "reason": "The answer determines whether execution nodes require durable local queues.",
  "answer_type": "boolean|choice|number|text|document_request",
  "status": "open|answered|deferred|unanswerable",
  "answer_claim_id": "optional uuid"
}
```

Questions should be ranked by expected decision impact.

## 8. Foundation Requirements

Before generating solutions, ArchetypeOS must compile the Genome and active claims into a
normalized set of foundation requirements.

```json
{
  "id": "uuid",
  "project_id": "uuid",
  "genome_snapshot_id": "uuid",
  "requirement_type": "hard_constraint|required_capability|quality_attribute|preference|optimization_goal",
  "domain": "deployment",
  "statement": "The production system must support operation without public cloud availability.",
  "priority": "must|should|could",
  "weight": 0.9,
  "veto_if_unsatisfied": true,
  "claim_ids": [],
  "verification_method": "architecture_inspection|test|benchmark|simulation|legal_review|human_review"
}
```

Hard constraints are evaluated before weighted scoring. A candidate that violates a hard
legal, safety, security, or deployment constraint is ineligible regardless of its
aggregate score.

### 9.1 Definition

A Foundation Candidate is a coherent, internally compatible set of decisions across all
foundation domains. It is not merely a technology stack.

## 10. Candidate Evaluation Model

### 10.1 Evaluation order

Candidates must be evaluated in this sequence: (1) hard-constraint eligibility; (2)
evidence sufficiency; (3) critical risk review; (4) required validation; (5) weighted
fitness scoring; (6) adversarial council review; (7) human selection. Weighted scores must
never override hard constraints.

### 10.2 Evaluation criteria

Initial criteria: requirement coverage; architectural coherence; security fitness;
privacy and compliance fitness; reliability fitness; operational feasibility; team
capability fit; cost fit; delivery speed; performance fitness; scalability fitness;
maintainability; testability; observability; reversibility; vendor lock-in; existing asset
reuse; migration complexity; evidence strength; residual uncertainty.

### 10.3 CandidateScore

```json
{
  "candidate_id": "uuid",
  "criterion": "operational_feasibility",
  "raw_score": 82,
  "weight": 0.8,
  "confidence": 0.74,
  "uncertainty_penalty": 8,
  "adjusted_score": 74,
  "rationale": "string",
  "supporting_claim_ids": [],
  "reviewer_agent_id": "uuid"
}
```

The UI should show the score vector, not only one overall number. A candidate with
Security 95 / Cost 40 / Team fit 35 / Overall 78 must not be presented simply as "78/100."
The shape of the tradeoff matters.

## 11. Validation Task Model

When evidence is insufficient, ArchetypeOS should prescribe validation rather than invent
certainty.

```json
{
  "id": "uuid",
  "candidate_id": "uuid",
  "title": "Benchmark local graph traversal latency",
  "validation_type": "prototype|benchmark|simulation|security_review|legal_review|user_test|vendor_check|integration_spike",
  "question": "Can the selected graph implementation answer the target traversal within 250 ms on the intended local hardware?",
  "method": "string",
  "success_criteria": [],
  "failure_criteria": [],
  "required_evidence": [],
  "blocking": true,
  "status": "proposed|approved|running|passed|failed|inconclusive",
  "result_claim_ids": []
}
```

Validation results become new evidence and may change the Genome or candidate ranking.

## 12. Foundation-Selection Lifecycle

Foundation selection should be implemented as a persisted state machine.

- **Stage 1 — Intake.** Inputs: project description, existing repositories, uploaded
  corpus, known goals, operator constraints. Outputs: project intake record, source
  manifest, initial sensitivity classification, declared analysis scope. Gate: all known
  sources are registered or explicitly marked unavailable.
- **Stage 2 — Corpus Freeze.** Version all sources, resolve repository commit references,
  hash source contents, create a Corpus Snapshot. Output: immutable analysis input set.
  Gate: every analysis result can reference a stable source version.
- **Stage 3 — Evidence Extraction.** Parse sources, extract fragments, create claims,
  assign truth layers, record authority domains, establish initial relationships. Outputs:
  evidence graph, extraction quality report, sensitive-source access policy. Gate: no
  material claim exists without provenance or an explicit `unsubstantiated` status.
- **Stage 4 — Current-State Reconstruction.** Scan repositories, run deterministic
  analysis, map runtime services, map architecture nodes/edges, examine tests/deployment/
  configuration/integrations. Outputs: current System Genome draft, current architecture,
  repository-to-capability map, observed risks and unknowns. Gate: current-state
  components must be backed by observed evidence.
- **Stage 5 — Intent Reconstruction.** Analyze PRDs, conversations, diagrams, decisions,
  communications; separate active from superseded intent; detect abandoned/conflicting
  concepts. Outputs: intended System Genome, intent chronology, contradiction inbox. Gate:
  conflicting material intentions are visible and not silently reconciled.
- **Stage 6 — Reconciliation.** Compare observed current system vs claimed intended system
  vs existing approved decisions. Outputs: drift findings, missing implementation,
  undocumented implementation, contradictions, superseded assumptions, open questions.
  Gate: critical conflicts are resolved, accepted as explicit risks, or marked as
  foundation blockers.
- **Stage 7 — Genome Classification.** Produce system traits, generate readable
  archetypes, score evidence coverage and confidence, identify foundation-shaping
  unknowns. Outputs: reviewed Genome snapshot, open-question queue, foundation requirement
  draft. Gate: all foundation-shaping traits have either sufficient evidence or an explicit
  uncertainty record.
- **Stage 8 — Foundation Requirement Compilation.** Translate evidence and Genome traits
  into hard constraints, required capabilities, quality attributes, preferences,
  optimization goals, verification obligations. Output: Foundation Requirement Set. Gate:
  hard constraints have clear source claims and verification methods.
- **Stage 9 — Candidate Generation.** Generate two to four genuinely distinct coherent
  candidates; each addresses every applicable foundation domain. Outputs: recommended
  candidate, credible alternative, conservative/reduced-complexity candidate, optional
  exploratory candidate. Gate: every candidate is internally coherent and explicitly
  addresses all hard constraints.
- **Stage 10 — Deterministic Eligibility Review.** Constraint evaluation, compatibility
  checks, dependency checks, licensing checks, deployment compatibility, security/privacy
  rules, existing-infrastructure compatibility. Outputs: eligible candidates; rejected
  candidates with exact reasons. Gate: no ineligible candidate proceeds to weighted
  selection.
- **Stage 11 — Engineering Council Challenge.** Applicable specialist agents review the
  same structured candidates. Suggested reviewers: Architecture Cartographer, Research
  Librarian, Technology Fitness Judge, Security Agent, Compliance Agent, Reliability/
  Operations Reviewer, Cost Reviewer, Domain Specialist, Human Factors/UX Reviewer,
  Verification Architect, Red-Team Skeptic. Each reviewer returns: findings, objections,
  unsupported assumptions, failure modes, required validations, blocking risks, conditional
  approval, evidence references. The current generic agent envelope can remain, but each
  agent needs a typed payload specific to its task. Generic arrays of findings and
  recommendations are insufficient for foundation adjudication. Gate: all blocking
  objections are resolved, accepted by an authorized person, or converted into validation
  tasks.
- **Stage 12 — Validation.** Execute prototypes, benchmarks, simulations, integration
  spikes, security reviews, legal reviews, cost checks, user workflow tests. Outputs: new
  evidence, candidate score changes, candidate rejection/promotion. Gate: every blocking
  validation task has passed or has an explicitly approved exception.
- **Stage 13 — Final Adjudication.** The Final Judge synthesizes the record but does not
  silently become the project authority. It produces: recommended candidate, reasons,
  remaining uncertainty, rejected alternatives, conditions of approval, required future
  reviews. Gate: human approval is mandatory for the selected foundation.
- **Stage 14 — Foundation Baseline.** The approved candidate becomes an immutable
  Foundation Baseline.

  ```json
    "target_genome_snapshot_id": "uuid",
    "version": "1.0",
    "status": "active|superseded|retired",
    "approved_decision_id": "uuid",
    "corpus_snapshot_id": "uuid",
    "approved_by": "operator",
    "approved_at": "timestamp",
    "effective_from": "timestamp",
    "review_triggers": [],
    "baseline_hash": "sha256"
  }
  ```

  The baseline contains: target System Genome, approved foundation elements, governing
  requirements, hard constraints, assumptions, risks, validation results, rejected
  alternatives, approval record, future review triggers. Gate: the baseline is immutable;
  changes create a new version.
- **Stage 15 — Execution Compilation.** Build Intelligence translates the baseline into
  implementation plans, work packages, dependencies, branch/worktree boundaries, acceptance
  criteria, verification procedures, agent assignments, human approval gates, rollback
  plans. The Foundation Baseline becomes the higher-order source from which one or more
  approved Decisions and plans are derived.
- **Stage 16 — Drift and Evolution Monitoring.** New evidence can trigger reevaluation
  when: repository architecture diverges from the baseline; a governing requirement
  changes; a critical source is superseded; a validation assumption fails; a dependency
  becomes insecure/unavailable; costs materially change; operational incidents invalidate
  an assumption; new legal/regulatory constraints appear. Outputs: drift event, stale
  decision warning, Genome delta, baseline review request, proposed replacement baseline.
  The Evolution Engine should evaluate whether the evidence changed enough to reopen
  foundation selection rather than automatically rewriting the baseline.

## 13. Selection Run State Machine

Suggested persisted statuses: `draft`, `intake_complete`, `corpus_frozen`,
`evidence_extracted`, `current_state_reconstructed`, `intent_reconstructed`, `reconciled`,
`genome_review`, `requirements_compiled`, `candidates_generated`, `eligibility_review`,
`council_review`, `validation_required`, `validation_complete`, `ready_for_selection`,
`selected`, `baselined`, `execution_compiled`, `monitoring`, `blocked`, `cancelled`,
`superseded`.

Every transition should record: actor, timestamp, previous state, new state, reason,
evidence generated, gate result, authority envelope (when applicable).

## 19. Decisions to Lock Now

1. The model is claim-centric, not document-centric.
2. Observed, claimed, inferred, and decided truth remain separate.
3. Current, intended, and target systems have separate Genome snapshots.
4. Repository DNA is evidence feeding the Genome, not the Genome itself.
5. Authority, confidence, relevance, and freshness are separate measures.
6. Evidence is immutable and versioned.
7. Conflicting evidence remains visible until explicitly resolved.
8. Hard constraints are evaluated before weighted scores.
9. Candidate scores cannot replace engineering judgment or human approval.
10. High uncertainty creates validation work, not invented certainty.
11. The foundation lifecycle is a persisted state machine.
12. Foundation approval creates an immutable versioned baseline.
13. Sensitive evidence is compartmentalized by policy.
14. Every foundation element must trace to claims, requirements, and verification.
15. Implementation plans derive from approved foundations, not directly from unreviewed prompts.

## 20. MVP Acceptance Criteria

The first complete vertical slice should demonstrate:

1. A user registers an existing repository.
2. The user uploads product notes, an architecture diagram, and a legal constraint.
3. ArchetypeOS freezes a corpus snapshot.
4. Repository inspection creates observed claims.
5. Uploaded documents create claimed requirements and constraints.
6. The system detects at least one contradiction between code and intent.
7. ArchetypeOS produces separate Current and Intended Genome snapshots.
8. Every foundation-shaping trait links to supporting or opposing claims.
9. ArchetypeOS compiles hard constraints and quality requirements.
10. Two meaningfully different foundation candidates are produced.
11. A candidate violating the legal constraint is deterministically rejected.
12. Specialist reviews produce structured objections.
13. An uncertain performance assumption creates a benchmark validation task.
14. Validation evidence changes the candidate comparison.
15. A human approves one candidate.
16. ArchetypeOS creates Foundation Baseline 1.0.
17. An implementation plan is generated from the approved foundation.
18. A later repository change produces a baseline-drift event.

That vertical slice would prove the entire concept without requiring every eventual
classification dimension, connector, or specialist agent.

## 21. Governing Principle

> No important recommendation should exist without a traceable chain from source, to
> claim, to system trait, to foundation requirement, to candidate decision, to validation,
> to human approval.

That chain is the core of ArchetypeOS as an Engineering Intelligence Platform.
