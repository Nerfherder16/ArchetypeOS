# LES-041 — a code-only PR (even one implementing an already-merged RFC) trips PR Guardian `missing-docs` + `missing-acceptance-evidence`; add a doc touch and an `## Acceptance Evidence` section, don't override

## Aliases

- Guardian BLOCK missing-acceptance-evidence — PR body has no '## Acceptance Evidence'
- code changed without documentation updates (but the RFC is already merged)
- do I override PR_GUARDIAN_OVERRIDE_DOCS / _ACCEPTANCE or add content

## Evidence

- PR #212 (`AOS-EVIDENCE-API-001`, the Evidence Spine HTTP routes) changed only runtime code +
  tests (`routes/evidence.py`, `schemas.py`, `main.py`, `services/evidence.py`, two test files) — no
  doc file — and PR Guardian BLOCKed with **two** findings:
  - `missing-docs`: "Implementation/runtime files changed without documentation updates."
  - `missing-acceptance-evidence`: "Code changed without an '## Acceptance Evidence' section."
- Contrast with the sibling PR #211 (models package), which PASSED Guardian: it bundled the RFC-0018
  doc + a `capability-map/layer-02.md` edit, so `missing-docs` never fired, and its body carried the
  evidence mapping. The API package carried no doc, so both checks fired. **The finding is correct**:
  a new API surface with no documentation and a PR body with no acceptance-evidence heading.
- Resolution (aligned, not override): (a) added an **`## Implementation Status`** section to the
  parent RFC (`docs/rfc/RFC-0018-…`) listing the shipped routes + guard→HTTP mapping — a real doc
  update that also keeps the RFC an accurate index of what's implemented; (b) added an
  **`## Acceptance Evidence`** section to the PR body mapping each acceptance criterion to its test;
  (c) recorded this lesson. Re-ran Guardian → PASS.

## Content

- Event: a pure implementation PR (routes over an already-merged, already-designed service) was
  BLOCKed for having no doc update and no acceptance-evidence section in the body.
- Root cause (by design): PR Guardian requires runtime-code changes to ship with (1) *some*
  documentation update and (2) an `## Acceptance Evidence` section in the PR body mapping criteria to
  verification. "The RFC already documents this" does not satisfy (1) or (2) — the checks look at the
  current diff and the current PR body, not merged history.
- Rules:
  1. A PR that changes runtime code needs a **doc touch in the same diff**. When the design already
     lives in a merged RFC, the natural touch is an **`## Implementation Status`** section on that RFC
     listing what this package shipped — real, useful documentation, not a rubber stamp. (Reserve
     `PR_GUARDIAN_OVERRIDE_DOCS` for changes that genuinely have no documentable surface.)
  2. Every code PR body needs an **`## Acceptance Evidence`** section mapping each acceptance criterion
     to a test / command / CI job. Add it when opening the PR, not after a BLOCK. (Reserve
     `PR_GUARDIAN_OVERRIDE_ACCEPTANCE` for the rare code change with no assertable acceptance.)
     **Exact format matters** (`tools/pr_guardian.py:303`): each criterion must be a **bullet line**
     (starts with `-` or `*`) that contains the literal token **`evidence:`** pointing at a test /
     command / CI job — e.g. `- C3 over HTTP → 422 — evidence: test_evidence_api.py`. A markdown
     **table** (`| Criterion | Evidence |`) does NOT satisfy it → the check downgrades from
     `missing-acceptance-evidence` to `empty-acceptance-evidence` (heading present, no qualifying
     line). Use bullets with `evidence:`, not a table.
  3. **Guardian reads the PR *body* on push, not on body-edit.** It snapshots
     `github.event.pull_request.body` at the `pull_request`/`synchronize` event, so editing the body
     alone never re-runs it against the new text — you must push a commit (even an empty/no-op one)
     AFTER the body edit, or better, edit the body FIRST then push, so the push event captures the
     corrected body. Ordering: update body → push; not push → update body (that races and re-blocks).
  3. Splitting a slice into a doc-bearing package (models: RFC + map) and a code-only package (API)
     means the code-only one will trip these unless you *plan* its doc touch + acceptance section up
     front. For every queued implementation-only package, pre-stage the RFC Implementation-Status
     edit and the PR-body Acceptance Evidence — same discipline LES-040 established for the capability
     map.
