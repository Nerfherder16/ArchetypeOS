# LES-042 — never redeclare an AuditMixin column in a subclass; it can import-OK locally but raises DuplicateColumnError in the compose/container import path

## Aliases

- sqlalchemy.exc.DuplicateColumnError: A column with name 'version' is already present in table X
- Docker Compose smoke test fails on alembic upgrade head but hermetic sqlite tests pass
- model imports fine locally but the API container won't start
- redeclared a mixin column (version/status/id/created_at/...) on a model

## Status

validated

## Owner

Opus orchestrator (cloud session)

## Evidence

- PR #215 (AOS-GENOME-MODELS-001) declared `version: Mapped[int] = mapped_column(Integer,
  default=1, nullable=False)` on `GenomeSnapshot(AuditMixin, Base)` — but `AuditMixin` already
  provides exactly that `version` column (`models.py:51`).
- **Hermetic tests + local import passed** (37 genome tests green under sqlite `create_all`;
  `python -c "import aos_core.models"`, `import app.main`, and `Base.metadata.sorted_tables` all
  succeeded locally), so the defect was invisible to the local gate.
- **The Docker Compose smoke test caught it**: the API container's startup (`alembic upgrade head`
  → `env.py` imports `aos_core.models`) raised
  `sqlalchemy.exc.DuplicateColumnError: A column with name 'version' is already present in table
  'genome_snapshots'` at declarative-mapping time, and the container exited 1.
- Same SQLAlchemy (2.0.36) and Python (3.12) in both places — so the difference is **import
  order/path nondeterminism**: redeclaring a mixin column is tolerated as an override in some
  import orders and collides in others (the container's boot path hits the colliding order).
- Fix: delete the redundant `version` column from `GenomeSnapshot`. The mixin still provides it
  (`'version' in GenomeSnapshot.__table__.columns` stays True), the genome "version N" semantics
  reuse it, migration `0029` (which lists `version` once per table) still matches → no drift, and
  all 37 tests stay green.

## The duplicate was in TWO places — and the migration one is the deeper trap

The redundant `version` existed both on the **model** (`GenomeSnapshot` redeclared the mixin column)
**and** in the **migration** (`0029` listed `sa.Column('version', ...)` explicitly *and* pulled it in
again via `*_audit_columns()`, which also contains `version`). Removing only the model column let
`env.py` import models, which then exposed the migration's own duplicate at
`op.create_table('genome_snapshots', ...)`.

**Why every local gate missed the migration bug:** the hermetic tests build schema with
`Base.metadata.create_all` — that is driven by the **model** metadata, so it NEVER executes the
Alembic migration. The migration's duplicated column is therefore invisible to `pytest`, to
`import aos_core.models`, and to `create_all`. The **only** gate that runs `alembic upgrade head` is
the **Docker Compose smoke test** (on Postgres). So a migration-only defect sails through the entire
local/hermetic suite and fails exclusively in compose-smoke.

**Reproduce migrations locally before pushing:** run the migration, not just `create_all` —
`DATABASE_URL="sqlite:///tmp.db" alembic upgrade head` catches the `DuplicateColumnError` at the
`create_table` step (a pure-Python schema error, dialect-independent). NB a later
`NotImplementedError: No support for ALTER of constraints in SQLite dialect` is expected and
**irrelevant** — it's a sqlite-only limitation of `op.create_foreign_key`/ALTER-ADD-CONSTRAINT;
compose-smoke runs on Postgres where ALTER ADD CONSTRAINT works, so getting *past* the duplicate
`create_table` is the signal that matters.

## Content

- Event: a model redeclared a mixin-provided column; local import + hermetic sqlite passed, but the
  compose-smoke container boot raised `DuplicateColumnError` and blocked the PR.
- Root cause: a subclass `mapped_column` with the same name as a mixin `mapped_column` is an
  *override* only some of the time — under the container's import order the two columns both attach
  to the one `Table`, which SQLAlchemy rejects. It is never correct to redeclare a mixin column
  identically.
- Rules:
  1. **Never redeclare an `AuditMixin`-provided column** (`id`, `status`, `version`, `created_at`,
     `updated_at`, `created_by`, `updated_by`, `meta`) on a model. The mixin provides it; if you need
     different semantics, use a differently-named column.
  2. **`import aos_core.models` passing locally does NOT prove the schema is container-safe.** The
     Docker Compose smoke test (real container import order + `alembic upgrade head` on Postgres) is
     the authoritative gate for model/migration import health — treat a green local import + green
     hermetic sqlite as necessary, not sufficient.
  3. When compose-smoke fails with `DuplicateColumnError` but local import passes, the culprit is a
     **redeclared mixin column** on the named table — grep the model for the column name and delete
     the redundant declaration; the fix is behavior-preserving (the mixin still supplies it) and
     needs no migration change.
  4. For any new model that reuses `AuditMixin`, diff its column list against the mixin's before
     committing — a fast `grep -nE "id:|status:|version:|created_at:" <model body>` catches it.
