# RFC Process

## Purpose

The RFC process keeps ArchetypeOS coherent as it grows.

Major capabilities, architectural changes, new engines, new agents, runtime changes, and governance changes should begin as RFCs before implementation.

## When An RFC Is Required

An RFC is required for:

- new engines
- new agents
- new runtime services
- new provider integrations
- governance changes
- security model changes
- data model changes
- major UI/UX changes
- autonomous action capabilities
- external system integrations

## RFC Lifecycle

```text
Draft -> Council Review -> Final Judge -> Accepted / Rejected / Deferred -> Implementation -> Validation -> Knowledge
```

## Required Sections

- Summary
- Problem
- Goals
- Non-goals
- Proposal
- Alternatives
- Evidence
- Risks
- Security impact
- Compliance impact
- Migration plan
- Acceptance criteria
- Open questions
- Final Judge verdict

## Naming

Use:

```text
docs/rfc/RFC-0000-title.md
```

## Principle

Important architectural change should be argued before it is coded.
