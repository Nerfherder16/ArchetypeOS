# Technology Fitness Engine

## Purpose

The Technology Fitness Engine compares technologies based on objective fit for the workload, not developer familiarity, popularity, or trendiness.

## Core Principle

Fitness over familiarity.

Developer comfort may affect rollout planning, but it must not determine the winning recommendation.

## Evaluation Domains

- workload fit
- performance
- concurrency
- ecosystem maturity
- operational complexity
- security posture
- compliance suitability
- cost
- hiring and maintenance
- deployment model
- scaling path
- migration effort
- long-term viability

## Technology Categories

- programming languages
- databases
- queues
- caches
- vector stores
- auth providers
- AI providers
- cloud providers
- frameworks
- infrastructure platforms
- observability tools

## Output

Each comparison should include:

- options considered
- scoring criteria
- score table
- recommendation
- why it wins
- tradeoffs
- migration plan
- risks
- confidence

## Optimization Modes

Recommendations may change depending on optimization target:

- fastest time to market
- lowest operating cost
- highest security
- maximum performance
- smallest team
- lowest maintenance
- longest product lifespan

## Principle

Ask: if we had no existing code and no prior comfort bias, what would we choose today?
