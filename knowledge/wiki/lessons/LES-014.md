## Content

- Event: the architecture spine graph models only directory containment; for a real monorepo the dominant structural signal — inter-package dependencies declared in manifests (and compose/service relationships, import graphs) — is absent, so the graph adds little beyond a folder tree.
- Source: AOS-21 reality test, self-found on a real external repo (predicted by the alpha run's architecture decision).
- Category: self-found-defect
- Lesson: **`contains` edges alone are not "architecture."** Once more than one repo — or one monorepo — is visible, the graph needs **manifest/dependency-derived edges** (workspace + dependency declarations), compose/service edges, and ideally import edges. This is the highest-value scanner/architecture follow-up now that a portfolio exists.
- Loop feed: **partially consumed by `AOS-ARCH-SEMANTICS-001` — the compose/service-edge half is delivered.** The scanner now parses each detected compose file (`yaml.safe_load`, tolerant) into `service` nodes and `depends_on` edges (both the list form `depends_on: [db, redis]` and the map form `depends_on: {db: {condition: ...}}`), and `run_scan` populates `RepositoryDNA.runtime_services`. **Still open: manifest/workspace dependency edges (`pyproject`/`package.json` resolution) and import-graph edges** — the monorepo half — remain a smaller follow-up.
- Status: open — compose/service edges shipped in `AOS-ARCH-SEMANTICS-001`; the row stays open for the remaining **manifest/dependency + import-graph edges**. Closed only when those land and this row cites them.
