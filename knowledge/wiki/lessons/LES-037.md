# LES-037 — An authority envelope is only real if the SERVER classifies the action, the approval is CONSUMED once, and it is BOUND to the thing it approved

## Aliases

- client-supplied action_class = no gate
- mark_executed was not a compare-and-swap (reusable approval)
- approval for one repo authorized another (unbound envelope)
- distillation minted a fresh pending request per retry
- AOS-AUTHORITY-HARDEN-001 closing lesson

## Status

closed

## Owner

Orchestrator (cloud session)

## Evidence

Verified on merged `main` implementing AOS-AUTHORITY-HARDEN-001 (WP4; consumes [[LES-033]]; builds on [[LES-036]] server-owned registry, [[LES-035]] operator auth):

- **The client classified the action.** `enqueue_job(action_class="read_only", ...)` — the class that decides whether approval is required was a caller-supplied parameter defaulting to the *safe* value, and no route derived it. A caller could submit a write/egress handler as `read_only` and sail through the gate. The evaluator was correct; the *input* to it was attacker-controlled.
- **The approval was reusable.** `mark_executed` was `ar.execution_state = "executed"` — a bare assign, no compare-and-swap, no commit. Two callers could both read `authorized` and both proceed; one authorized envelope could drive many executions.
- **The approval was unbound and immortal.** `ActionRequest` had no `repository_id`/`job_id`/`expires_at`; an approval for one repository could be replayed against another, forever, with no execution trace of what it authorized.
- **Distillation minted a fresh pending request every retry** and hardcoded `sensitivity="public"`, so a private repo's content egressed without approval and approvals piled up unconsumed.

## Content

- **Event:** the authority envelope was structurally present (request → approve → gate) but the classification was client-supplied, the consume was not atomic, the approval was unbound/immortal, and the one direct-egress path (distillation) leaked private data and spammed pending requests.
- **Category:** self-found-defect (found while hardening the envelope, verified against source).
- **Lesson (transferable):**
  1. **The SERVER must classify the action; never trust the client's action class.** Derive `action_class` (and sensitivity) from a server-owned registry keyed by the operation type. The client says *what* it wants done; the server decides *how dangerous that is* and therefore whether approval is required. A safe default parameter the caller can override is not a gate.
  2. **A one-use approval is a compare-and-swap, not an assignment.** `UPDATE ... SET state='executed' WHERE state='authorized'` and check `rowcount==1`; the DB row lock makes concurrent consumers resolve to exactly one winner. `synchronize_session=False` avoids SQLAlchemy's Python-side eval comparing a naive stored datetime against an aware `now`. Consume inside the same transaction as the effect it authorizes (e.g. job creation) so the approval and the action commit together — and consume AFTER a direct action succeeds, never on failure, or a raised action burns the approval.
  3. **Bind the approval to the thing it approves, and expire it.** Persist `repository_id`/`payload_digest` (+ an execution FK to the job that consumed it) and check them at consume time — an approval for one target/payload must not authorize another. Add an `expires_at` so a stale approval cannot be replayed indefinitely.
  4. **Direct high-impact actions need approve-and-resume, not a fresh request per try.** A blocked call returns the pending id; the operator approves; the retry passes that id, which is validated + consumed — it does not mint a new pending envelope each attempt. And derive the action's sensitivity from the resource's persisted policy (a private repository requires approval to egress), not a hardcoded `public`.
  5. **Reject unknown operation types before persistence.** An unregistered `job_type` should 4xx at origination, not get durably queued and discovered unroutable later.
- **Loop feed:** the server-owned classification + `consume_action` are the seam for extending the envelope to council/research egress and any future write/deploy path (a follow-up); the bound/expiring `ActionRequest` is the audit substrate for the operator approvals view.
- **Status:** closed — consumed by AOS-AUTHORITY-HARDEN-001 (branch `claude/aos-authority-harden-001`): server-owned `action_class`, unknown-type rejection, atomic one-use `consume_action` CAS + job-link, `repository_id`/`job_id`/`expires_at` binding, distillation approve-and-resume + repo-sensitivity policy, migration 0025, PostgreSQL one-winner concurrency test.
