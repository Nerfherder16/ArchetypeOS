# LES-035 — The route that GRANTS authority must itself be authenticated, and "empty token = open" is not fail-closed on a deployed profile

## Aliases

- unauthenticated node enrollment / authority approval
- operator plane had no auth dependency
- soft-token gate leaves deployed profiles open
- AOS-AUTH-BOUNDARY-001 closing lesson

## Status

closed

## Owner

Orchestrator (cloud session)

## Evidence

Verified on `main` @ `267f50f` implementing AOS-AUTH-BOUNDARY-001 (WP2 of the post-AOS-REVIEW-002 wave; consumes [[LES-033]]):

- **The grant path was anonymous.** `POST /nodes/enroll` (`routes/nodes.py:41`) — the *only* path that grants a node `write_access`/`max_sensitivity` and issues its credential — had only `Depends(get_db)`. `POST /authority/actions/{id}/authorize` and `/reject` (`routes/authority.py:75,83`) were likewise unauthenticated, and the approver was a hardcoded literal `"operator"`. The heartbeat route *was* correctly gated by `require_node_token`, which made the gap easy to miss: the cheap, high-frequency action was protected while the rare, high-impact grant was open.
- **"Empty token = open" is fine for a tailnet but wrong for a deployment.** `audit_heartbeat_token`/`connector_write_token` defaulted `""` and every gate was `if settings.<token> and …`, so an unset secret meant *open, unconditionally* — with no distinction between "local single-operator box" and "deployed multi-tenant profile." The shipped `docker-compose.yml` set neither secret, so those gates shipped open.
- **Comparisons were not constant-time.** The connector and audit gates used `x != settings.token`; only the node-token path used `secrets.compare_digest`.

## Content

- **Event:** the node/connector/authority registries were made real in the prior wave, but the *routes that mutate authority* (enroll, approve/reject) were never given an auth dependency, and the token gates that did exist failed *open* on a deployed profile.
- **Category:** self-found-defect (found while hardening the control plane, verified against source).
- **Lesson (transferable):**
  1. **Authenticate the route that grants or changes authority, not just the one that uses it.** A protected heartbeat next to an unprotected enrollment is a false sense of security — the enrollment is where privilege is minted. When auditing an auth boundary, enumerate every route that *writes* policy/credential/approval state and confirm each has the gate, rather than spot-checking the obvious high-traffic one.
  2. **"Token unset = open" must be an explicit, named mode, and a deployed profile must fail closed.** Split "local/tailnet dev" (open, but logged so it is never silent) from "deployed" (no secret → refuse, `503`). A single reusable dependency (`require_operator`) with `operator_token` + an explicit `auth_dev_mode` flag, and a compose file that sets `AUTH_DEV_MODE=false`, turns "absent secret" from *silently open* into *loudly closed*.
  3. **Record the actor on every authority mutation.** A hardcoded `approver="operator"` makes approvals unattributable; thread a real principal (an `X-Operator-Id`, default `operator`) from the auth dependency into `updated_by` so the approval history names who approved.
  4. **Constant-time compare every secret, and preserve bootstrapping.** Use `secrets.compare_digest` uniformly. When binding an action to a credential (a node can only re-register itself with its token), keep the first-contact path open for a credential-less new node, or you deadlock enrollment.
- **Loop feed:** `require_operator` is reused by AOS-AUTHORITY-HARDEN-001 (WP4) for the approve-and-resume flow, and the fail-closed-deployed-profile pattern feeds the remaining packages that add secrets.
- **Status:** closed — consumed by AOS-AUTH-BOUNDARY-001 (branch `claude/aos-auth-boundary-001`): `require_operator` on enroll/authorize/reject + new rotate/revoke routes, fail-closed deployed compose, constant-time gates, recorded approver, node-token re-registration binding, with a security test suite.
