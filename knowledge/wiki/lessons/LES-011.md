# LES-011 — compose-smoke builds an explicit service list, not "all services"

## Aliases

- unbuilt compose service
- scheduler CI gap

## Status

validated

## Owner

Chief Architect / Orchestrator

## Evidence

- AOS-SCHED-001 review: the new `scheduler` compose service was defined in `docker-compose.yml` but the `compose-smoke` CI job builds/starts an explicit list (`docker compose build api worker web`; `up -d ... worker web`). `docker compose config` validates the file but does not build images, so the scheduler Dockerfile + boot would have shipped CI-unverified. Caught in Orchestrator review; ci.yml updated to include `scheduler` in both the build and up steps.

## Content

- Event: a new compose service was added without adding it to the compose-smoke build/up lists; only `docker compose config` (validation, not build) would have run against it.
- Source: AOS-SCHED-001 review remediation, self-caught.
- Category: review-remediation
- Lesson: the `compose-smoke` job enumerates services explicitly — any NEW compose service must be added to both the `docker compose build ...` and `docker compose up -d ...` steps, or its image build and boot are silently unverified by CI. Echoes LES-006 (a check that never exercises the new thing is a silent gap). Candidate hardening: switch compose-smoke to `docker compose build` (no args → all services) so new services are covered automatically.
- Loop feed: ci.yml fix in this package; guardian/CI-hardening candidate (build-all) noted.
- Status: closed — consumed by the AOS-SCHED-001 ci.yml fix + this recorded practice.
